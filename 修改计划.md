## 零、设计原则（讨论确认）

1. **因果潜力**：一件事可以导致连续的因果影响 / 介入已有事件线 / 催生新事件。**大小不重要，有没有潜力才重要**
2. **标签 vs 场景**：同一个场景（如擂台赛），成为选手→触发，误被波及→触发，观战→不触发。**不按场景类型定规则，全凭 AI 对叙事上下文的判断**
3. **四前缀状态**：`介入`(新因素进入)、`退出`(主角离开)、`结束`(事件终结)、`延续`(无新分支的连续进行)
4. **LLM 不分关键NPC/龙套**：不影响本系统——`<情节事件>` 是 AI 对叙事节奏的自我感知，不是变量系统的输入
5. **系统不主动操控**：计数器、事件记录、分档文本——全是描述性的。AI 看到上下文自由发挥
6. **延续不阻塞介入**：延续回合 AI 仍可输出介入标签，代码不拦截
7. **多标签处理**：同回合输出多个 `<情节事件>`，取其中产生**最低计数值**的那个（最冲击性的标签主导节奏重置）

### 计数器行为总表

| AI 输出 | 平静计数变化 | 说明 |
|---------|-------------|------|
| `<情节事件>介入：XXX</情节事件>` | 归零 0 | 新因果链进入 |
| `<情节事件>退出：XXX</情节事件>` | 归零 0 | 主角离开事件 |
| `<情节事件>结束：XXX</情节事件>` | 归零 0 | 事件本身终结 |
| `<情节事件>延续：XXX</情节事件>` | +1（可配） | 有事件但无进展，慢涨 |
| 无标签 | +2（可配） | 纯过渡/间歇，比延续快一倍 |

延续和无标签的计数值增速不同，区别在增量配置。AI 从 `情节事件记录` 和活跃事件行感知当前是否有事件在进行。

**范围：0 ~ 32**。到达 32 封顶不再上涨。无标签增量默认 2，延续增量默认 1（均可配置）。

**多标签例子**：同一回合 `介入新追兵(0)` + `延续旧追兵(+1)` → 取最小值 0。

## 一、数据模型

### 1. 叙事状态结构（独立顶级字段）

```ts
export interface 叙事状态结构 {
  平静计数: number;         // 0~32，连续无剧情波折的回合数
  情节事件记录: string[];    // 最近 30 次介入/退出/结束记录（供 AI 上下文参考）
}

export interface 叙事平静值配置结构 {
  启用: boolean;              // 总开关
  无标签增量: number;          // 默认 2
  延续增量: number;            // 默认 1
  上限: number;                // 默认 32
  最低触发阈值: number;        // 默认 12，低于此值不注入任何文本
  阈值文本: string[];          // 从最低触发阈值到上限等分若干段，每段一个文本（默认 6 段）
}
```

> **等分区段设计**：不再用 `12_14`/`15_18` 等硬编码键名。程序根据 `最低触发阈值`（默认 12）和 `上限`（默认 32）自动等分 `阈值文本` 数组长度份区段。例如 6 段时每段宽度 = (32-12)/6 ≈ 3.33，计数落入哪段就用哪段文本。用户只需调"最低触发阈值"和"上限"，区段自动等分。

### 2. 配置结构

```ts
// 在 游戏设置结构 中，取代之前的简单 boolean
叙事平静值配置?: 叙事平静值配置结构;
```

写档/读档时序列化整个配置结构。

### 3. 数据流向

```
叙事状态结构
  ├── 创建（默认值：平静计数=0，记录=[]）
  ├── 持久化（存档/读档时序列化）
  ├── 快照（回档时保存）
  ├── 注入 AI 上下文 → 系统提示词构建
  ├── AI 响应解析 → 提取前缀和内容
  ├── 回合后更新 → 无标签/延续→+1, 介入/退出/结束→归零
  └── 本地事件联动 → 从当前环境/世界状态提取附近局部细节
```

## 二、提示词设计

### 原则

| 交互点 | 做什么 | 不做什么 |
|--------|--------|---------|
| 上下文（每回合给 AI） | 呈现当前平静程度 + 当前活跃事件 + 附近细节 | 不加"你应该..."的指令，不写"叙事平淡/节奏"等元叙事措辞 |
| 标签说明（格式协议） | 告诉 AI `<情节事件>` 存在 | 不暗示 AI"该用了"或"别用了" |
| 变量生成（variable model） | 不修改 | 不涉及叙事状态字段 |
| 规划分析（planning） | 可看到平静信息 | 不加"主动制造波折"的指令 |

### 架构约束（防止"世界围着主角转"）

叙事平静值系统、世界演变、剧情规划三者的职责边界：

| 系统 | 负责 | 不负责 |
|------|------|--------|
| 叙事平静值 | 提示"新事件可以展开了"，不指定发生在谁身上 | 生成"主角身边必须出事"的指令 |
| 世界演变 | 生成"远处正在发生的事"（城东灭门、西山劫镖） | 强制主角参与 |
| 剧情规划 | 跟踪现有 NPC 独立意图的推进 | 把所有阴谋指向主角 |

配合方式：平淡值 ≥19 时 AI 可考虑推进新的事件方向，但不一定是"突然有人闯入茶摊打搅主角"。未来可对世界事件系统做因果律改造，让世界演变中的事件自动关联主角因果链，届时平淡值可与世界事件联动。

### 2.1 上下文注入 — 叙事状态 + 本地事件联动

**位置**：`systemPromptBuilder.ts`，构建 `系统提示词` 时，随 `statePayload` 传入

**格式**：在 AI 感知世界状态的段落中，追加以下内容（仅在启用且计数 >= 6 时）。

无前缀、无元叙事措辞，像一行自然观察：

```
当前：灭门案调查（仍在进行）
周围很安静，远处似乎有些动静。（2 个方向）
```

或用户/Mod 自定义文本。全程不使用"叙事""节奏""平淡""注入"等术语。

**阈值注入表（用户/Mod 可覆盖每档文本）：**

| 平静计数 | 默认文本 | 说明 |
|---------|---------|------|
| < 12 | （不注入） | 低于最低触发阈值 |
| 第 0 段 | `一切如常。` | 最低触发阈值附近 |
| 第 1 段 | `一切如常，有些细节可以留意。（1 个方向）` | |
| 第 2 段 | `一切如常，远处似乎有些变化。（2 个方向）` | |
| 第 3 段 | `太长时间没有变化了，或许该有些新的事。（3 个方向）` | |
| 第 4 段 | `风平浪静得太久了，可以有些新的动向了。（4 个方向）` | |
| 第 5 段 | `一切如常，只是似乎有什么事要发生了。` | 接近上限 |

> 区段由 `最低触发阈值`（默认 12）和 `上限`（默认 32）等分 `阈值文本.length` 份自动计算，不硬编码键名。

**联动说明**：`情节事件记录` 独立于世界事件系统。两者维度不同——前者跟踪 AI 对叙事节奏的标签标记，后者跟踪世界客观时间线。在世界事件引入因果律改造后，可考虑联动。目前阶段不耦合。

**实现**：

```ts
const 构建叙事状态文本 = (config: 叙事平静值配置结构, ns: 叙事状态结构): string => {
  if (!config?.启用) return '';
  const 计数 = ns.平静计数 ?? 0;
  const 下限 = config.最低触发阈值 ?? 12;
  const 上限 = config.上限 ?? 32;
  const 文本列表 = Array.isArray(config.阈值文本) ? config.阈值文本 : [];

  // 低于最低触发阈值不注入
  if (计数 < 下限) return '';

  // 获取当前活跃事件（最近一个介入且无对应退出/结束的事件）
  const 活跃事件 = 获取当前活跃事件名(ns.情节事件记录);

  // 等分区段选档：将 [下限, 上限] 等分 文本列表.length 份，计数落入哪段就用哪段文本
  let text = '';
  if (文本列表.length === 0 || 计数 >= 上限) {
    text = 文本列表[文本列表.length - 1] || '一切如常，只是似乎有什么事要发生了。';
  } else {
    const 段宽 = (上限 - 下限) / 文本列表.length;
    const 段索引 = Math.min(文本列表.length - 1, Math.floor((计数 - 下限) / 段宽));
    text = 文本列表[段索引] || '';
  }

  const parts: string[] = [];
  if (活跃事件) {
    parts.push(`当前：${活跃事件}（仍在进行）`);
  }

  if (text) parts.push(text);
  return parts.join('\n');
};

const 获取当前活跃事件名 = (情节事件记录: string[]): string | null => {
  for (let i = 情节事件记录.length - 1; i >= 0; i--) {
    const entry = 情节事件记录[i];
    const im = entry.match(/介入[：:]\s*(.+)/);
    if (!im) continue;
    const 事件名 = im[1].trim();
    const later = 情节事件记录.slice(i + 1);
    const hasExit = later.some(e => e.includes(`退出：${事件名}`) || e.includes(`结束：${事件名}`));
    if (!hasExit) return 事件名;
  }
  return null;
};
```

### 2.2 标签格式说明 — AI 知道这个标签存在

**位置**：`prompts/core/format.ts`，第 1 节"顶层标签规则"（line 17）

**追加**：在"可选顶层标签"列表（line 19）中加入 `<情节事件>`

> **代码库核对**：format.ts line 31 规定"顶层每个最多出现 1 次"。`<情节事件>` 设计为每回合可输出多个（同回合多标签取最小计数值）。需明确 `<情节事件>` 是**可重复标记标签**，不受"最多 1 次"约束。在 line 19 追加后，需在 line 31 附近补注："`<情节事件>` 不受此限，每回合可输出多个。"

**追加说明段落**：

```
- <情节事件> 是可选标记标签，格式为：<情节事件>前缀：事件名</情节事件>。
  前缀有四种：
    · 介入：新的因果链进入叙事（新敌人、新线索、新地点、新人物）
    · 延续：同一事件的连续进行中，无新因果分支
    · 退出：主角明确放弃/脱离该事件，事件本身未结束（溜号、脱战、放弃追查）
    · 结束：事件本身终结/解决（结案、杀光追兵、关系定型）
  ⚠ 同一场面连续进行时，每回合输出相同的「延续：XXX」。
  ⚠ 只有场景内出现新因果分支才「介入」，只有事件终了才「结束」，只有主角离开才「退出」。
  ⚠ 退出 ≠ 场景切换。短暂离开（睡觉、赶路、吃饭、临时离开场景）后又回来继续同一事件，应使用「延续」而非「退出→介入」。
  ⚠ 延续期间也可以介入新事件——输出新标签即可。
  ⚠ 每回合结束前请评估当前叙事节奏，至少输出一个 <情节事件> 标签（延续或介入/退出/结束均可）。连续多回合无标签可能被系统判定为平淡。
  无因果潜力的回合（原地徘徊、重复描写）不输出此标签。
  本标签不附带"应该怎么做"的后续——它只是标记。
```

**典型示例**：

```
调查灭门案:
  <情节事件>介入：武陵山庄灭门案</情节事件>
  <情节事件>延续：武陵山庄灭门案</情节事件>
  <情节事件>退出：武陵山庄灭门案</情节事件>   ← 查一半跑了
  ...
  <情节事件>介入：武陵山庄灭门案</情节事件>   ← 又回来查
  <情节事件>结束：武陵山庄灭门案</情节事件>   ← 破案了

连续战斗:
  <情节事件>介入：仇敌追杀</情节事件>
  <情节事件>延续：仇敌追杀</情节事件>
  <情节事件>介入：追兵突袭</情节事件>         ← 新因素
  <情节事件>结束：追兵突袭</情节事件>         ← 追兵死光
  <情节事件>退出：仇敌追杀</情节事件>         ← 主角跑了

偷听密谋:
  <情节事件>介入：密谋会议</情节事件>
  <情节事件>延续：密谋会议</情节事件>
  <情节事件>退出：密谋会议</情节事件>         ← 溜了，会还在开
```

**注入机制**：通过现有的 `获取输出协议提示词()` 自动注入。不需要新建 prompt 文件。

## 六、世界事件距离化改造（ver 7 新增）

### 6.0 背景与根因定位

实测现象：世界演变生成的世界事件，少说一半最终都被 AI"靠"到主角附近——主角在西山砍柴，城东灭门案也会演成"恰好路过听到呼救"。

根因经完整链路排查，定位在六个环节叠加，不是单一提示词问题：

1. **数据层缺距离标记（最核心）**：`待执行事件 / 进行中事件 / 已结算事件` 没有"离主角多远、主角该直接遇到还是只该听说"的字段。`关联地点[]` 只是字符串数组，LLM 不擅长隐式空间比对，默认倾向"既然注入了就和主角有关"。
2. **主剧情 COT Step5 用"触发"措辞**：`确认本回合是否存在适合触发的当前事件`——"触发"有"让它发生"的语义倾向，且没给"该上台 vs 只做背景"的判断标准。
3. **防全知协议的反向激励（悖论）**：锁定玩家视角 + 不得描写玩家不在场场景，导致 AI 倾向把事件拉到主角面前（阻力最小路径），而非构思传闻传播链。本意防全知，副作用是强拉事件。
4. **世界演变补位规则偏向主角**：补位优先"既有旧线 > 已成立人物下一段动作"，而既有旧线几乎都是主角相关的。
5. **规划分析等权呈现全部事件**：无距离过滤，AI 选"最相关"= 主角相关。
6. **LLM 固有主角中心化倾向**：被整个系统提示词的主角框架放大。

**结论**：单靠数据字段不够。必须在数据层 + 主剧情 COT + 防全知协议 + 规划分析 + 世界演变 五处同步改造，才能打破路径最小阻力。

**核心洞察**："世界围着主角转"不是世界演变 AI 的错，而是主剧情 AI 在消费世界事件时，没有足够的约束阻止它走阻力最小路径。因此改造重心应在**主剧情消费侧**（COT + 防全知 + 注入分档），而非只在生成侧（世界演变）。

### 6.1 设计原则

1. **距离字段是建议，不是锁**：`主角参与度` 告诉 AI 该事件与主角的关联强度，但 AI 仍可基于正文事实调整——只是默认不再"无依据地拉到主角面前"。
2. **三档注入**：主剧情系统提示词按 `远处 / 附近 / 当场` 分三档呈现世界事件，而非平铺。远处事件只给"传闻"层信息密度。
3. **传闻链是合法感知方式**：明确告诉 AI，"听说 / 传闻 / 公告 / 他人转述 / 消息流传"是远处事件的合法获知渠道，不算全知视角违规——它正是远处事件进入主角认知的正确渠道。
4. **不阻塞世界独立性**：远处事件可以在世界演变中独立推进、独立结算，不需要主角参与也有因果来源。
5. **与叙事平静值系统联动（可选）**：平淡值 ≥ 19 时，世界演变可优先推进"附近 / 远处"事件，不推进"当场"事件（避免强拉主角）。

```
  · 大规模冲突、他方势力变动、远地大事 → 0.1（远处）
  · 区域性事件 → 0.2（远处，除非主角当前就在该区域）
  · 主角约定事件（如三天后的决斗）→ 0.9（当场）
- 若事件关联地点与主角当前地点链重合，参与度可上调 0.3，但不超过 1.0。
- 参与度一旦设定，后续回合默认不主动上调；只有正文事实坐实主角已卷入时，才在世界演变中更新。
```

`worldEvolutionCot.ts` Step5 追加：

```
- 事件补位时，新事件必须带"主角参与度"字段；远处事件（≤0.2）的补位不得依赖主角行动才能成立。
```

**参与度计算策略（AI 标记为主，程序化兜底为辅）**：

- 世界演变 AI 生成事件时**必须**标记 `主角参与度`（在上文提示词中强制要求）。
- 若 AI 漏标，`规范化世界状态()` 兜底：按 `关联地点` 与主角当前地点链做简单匹配（重合→0.6，同大地点→0.4，否则→0.2）。
- **不做程序化自动覆盖**：不强制用规则覆盖 AI 标记值，因为程序化规则无法处理复杂间接因果（如"主角救了张三→张三所在势力内乱→主角与此内乱有间接因果"），AI 的因果理解能力优于固定规则。
- 程序化规则仅作为 `?? 兜底`，不作为 `= 覆盖`。

#### 6.3.7 `prompts/runtime/worldEvolutionCot.ts` Step0 承接 vs 推导区分 + 正向表述

> Claude 深挖分析（修改问题.md 7.2-7.3）指出 worldEvolutionCot.ts Step0 将"本回合前台已发生事实"作为推理依据，与"背景演变不是台前跟拍"的设计声明矛盾。但**完全删除前台事实会破坏世界状态一致性**（主角杀了人，世界演变需更新活跃 NPC 列表）。正确方案是区分"承接已发生后果"与"以主角行动为推理起点"。

**Step0 证据清单追加约束**（不删除前台事实，但限定其用途）：

> **提示词风格更新（2026-06-30 搜索结论）**：基于 2026 年研究（arXiv:2602.04294），ToP（任务规则）+ 独立正反例列表会降低 LLM 服从率 -6.6%（最多 -21.2%），因为 few-shot 列表拉长上下文、把指令推到低注意力位置。而 RRP（角色身份 + 规则嵌入）是 2025 年 CPDC 基准测试最优模式（arXiv:2509.00482）。因此本约束采用**嵌入式单行示例**（正反例嵌入规则句内），不独立成正反例列表。

```
- 作为后台世界维护者，你承接前台已发生的后果（更新活跃 NPC 列表、迁移事件状态、刷新势力关系），但新事件方向只从世界客观张力中推导（势力矛盾、时间到期、旧线发酵），不从主角当前位置或行动中推导。例：主角杀人→更新活跃 NPC 列表（承接）；主角在西山→不因此生成西山事件（推导边界）。
```

**负面表述改正向**（Step1 更新必要性审计）：

原：
```
- 若当前信息仍停留在台前场景跟拍、当前在场角色的即时承接或尚未沉淀的现场互动，就先排除出本链路
```

改为：
```
- 只处理已转化为后台可持续状态的信息：远端余波、势力动作、事件窗口、环境变化或未来执行窗口。尚未沉淀的台前场景互动不进入本轮审计。
```

> 不大规模改写 worldEvolutionCot.ts 的"当前"时间锚定语言。原因：worldEvolutionCot.ts 中的"当前"多数指"当前世界状态"而非"主角当前状态"（如"当前世界维护使用的是新结构"），AI 读了完整提示词能理解这个区别。大规模改写有破坏世界演变与主剧情承接能力的风险，且收益不确定。

### 6.4 与叙事平静值系统的联动（阶段 2，在平淡值系统基础功能完成后再做）

联动分三层，各自独立可单独实施：

#### 6.4.1 数据层联动——分档注入中读取远处事件

**位置**：`hooks/useGame/systemPromptBuilder.ts` 的 `构建叙事状态文本()` 函数

当 `平静计数 >= 19` 时，从 `规范化世界状态` 读取参与度 ≤ 0.5 的待执行/进行中事件，注入为"远处/附近有变化的线索"：

```ts
// 在 构建叙事状态文本() 中，当计数 >= 19 时追加
if (计数 >= 19 && world) {
  const 远处线索 = [
    ...取数组(world?.待执行事件).filter(e => (e?.主角参与度 ?? 0) <= 0.2),
    ...取数组(world?.进行中事件).filter(e => (e?.主角参与度 ?? 0) <= 0.2)
  ].slice(0, 2); // 最多 2 条，避免信息过载
  if (远处线索.length > 0) {
    parts.push('远处动向：' + 远处线索.map(e => `${取文本(e?.事件名)}（${取文本(e?.当前进展) || 取文本(e?.事件说明)}）`).join('；'));
  }
}
```

**效果**：平淡值高时，AI 上下文中会出现远处事件的线索，配合距离化约束"远处只能传闻"，远处事件以传闻形式呈现。

#### 6.4.2 提示词层联动——阈值文本与距离化约束协同

**无需额外代码改动**：阈值文本已在 `叙事平静值配置结构` 中定义，距离化约束已在 COT Step5 / 防全知 5.1 / format 5.5 中落地。两套提示词天然协同。

#### 6.4.3 世界演变联动（可选）——平淡值高时优先推进远处事件

```ts
// worldEvolutionWorkflow.ts 中，世界演变触发前
if (叙事平静值配置?.启用 && 叙事平静值.平静计数 >= 19) {

}
```

**效果**：平淡值高时，远处事件有进展（如"听说城东抓到凶手了"），但不会强制主角参与。

### 6.8 与 Claude 深挖分析的差异说明

Claude 在 `修改问题.md` 7.1-7.10 提出了详细的根因分析。本计划采纳其大部分结论，但对以下三项做了修正：

#### 6.8.1 不采纳"删除 worldEvolutionCot.ts Step0 中的'本回合前台已发生事实'"

Claude 7.3 + 清单 A 建议删除该证据项。但**完全删除会破坏世界状态一致性**：

- 主角杀了某 NPC → 世界演变需更新活跃 NPC 列表移除该 NPC → 需要前台事实作为依据
- 主角与某势力冲突 → 世界演变需更新势力关系网 → 需要前台事实作为依据
- 主角解决了某事件 → 世界演变需将事件从进行中迁入已结算 → 需要前台事实作为依据

**正确方案**：保留前台事实作为承接依据，但追加"承接 vs 推导"区分（6.3.7）：
- 合法用途：承接已发生后果（更新列表、迁移状态、刷新关系）
- 禁止用途：以主角行动为起点推导新事件方向

#### 6.8.2 不采纳"大规模改写 worldEvolutionCot.ts 时间锚定语言"

Claude 7.2 + 清单 A 建议将"当前回合""当前状态""当前章节"改为"世界客观时间线""已确认事件"。

但 worldEvolutionCot.ts 中的"当前"多数指"当前世界状态"而非"主角当前状态"（如"当前世界维护使用的是新结构"）。Claude 的统计"49.4% 的行包含主角/玩家/你/当前"存在偏差：很多"主角"是禁止性规则（"主角不得进入活跃 NPC 列表"），很多"当前"指世界状态。

大规模改写有破坏世界演变与主剧情承接能力的风险，且收益不确定。**只采纳负面表述改正向**（6.3.7 Step1），这是低风险改进。

#### 6.8.3 不采纳"程序化自动计算参与度覆盖 AI 标记"

Claude 清单 E 给出了程序化计算函数 `计算主角参与度()`。但程序化规则无法处理复杂间接因果（如"主角救了张三→张三所在势力内乱→主角与此内乱有间接因果"）。

**正确方案**（6.3.5）：AI 标记为主，程序化兜底为辅。AI 的因果理解能力优于固定规则；程序化规则仅在 AI 漏标时作为 `?? 兜底`，不作为 `= 覆盖`。

### 6.10 提示词风格设计依据（2026-06-30 网络搜索结论）

> 本节记录提示词措辞风格的设计依据，基于 2025-2026 年 LLM 约束服从率研究。

#### 关键研究结论

| 研究 | 核心发现 | 对本计划的影响 |
|------|---------|--------------|
| arXiv:2602.04294 (2026) | ToP（任务规则）+ 独立 few-shot 列表 → 服从率 **-6.6%**（最多 -21.2%）；RoP（角色身份）+ few-shot → **+2.0%**（最多 +4.5%） | Step0 承接vs推导约束改用嵌入式单行示例，不独立成正反例列表 |
| arXiv:2509.00482 (2025, CPDC) | RRP（角色身份 + 显式规则集）得分最高（~0.57-0.60 vs 基线 0.519） | `worldEvolutionCot.ts:8` 已有角色框架（"你的职责是背景演变"），新增约束应依附此框架 |
| arXiv:2405.18027 (2024, TimeChara) | 明确的空间/时间知识边界约束在创意写作中有效，减少"全知视角"违规 | 验证了三档分档注入 + 强拉禁令的设计方向正确 |
| arXiv:2510.22251 (2025) | 前沿模型原生指令跟随能力提升，过度 elaborate 的约束可能适得其反（"handcuffs"效应） | 提示词改动保持"约束追加"而非"重写"，避免过度工程化 |

#### 设计原则落地

1. **嵌入式示例 > 独立列表**：所有正反例以单行嵌入规则句内（如"例：主角杀人→更新列表（承接）；主角在西山→不因此生成事件（推导边界）"），不独立成 `✅ 合法 / ❌ 禁止` 列表。
2. **依附角色框架**：`worldEvolutionCot.ts` 已有"你的职责是背景演变，不是台前跟拍"的角色定义（RoP 基础），新增约束使用"作为后台世界维护者，你..."的措辞依附此框架，而非纯 ToP 规则堆叠。
3. **传闻链示例单行嵌入**：COT Step5 中"传闻层也要写出传播渠道（谁说的、在哪听到、公告贴在哪）"以单行嵌入，不独立成示例列表。
4. **约束追加 > 重写**：所有提示词改动是在现有 Step 末尾追加规则，不改动原有推理链结构——避免破坏既有 RRP 框架的稳定性。

#### 不采纳的方案

| 方案 | 不采纳原因 | 研究依据 |
|------|-----------|---------|
| Step0 独立正反例列表 | ToP + few-shot 独立列表降低服从率 | arXiv:2602.04294 |
| 大规模改写 COT 时间锚定语言 | 过度 elaborate 约束对前沿模型可能适得其反 | arXiv:2510.22251 |
| 传闻链独立示例库 | 上下文稀释，把指令推到低注意力位置 | arXiv:2602.04294 |
