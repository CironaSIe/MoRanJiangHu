# 创意工坊与小说分解系统审计

## 一、创意工坊模块系统

### 1.1 模块发布/下载/注入流程

**文件**: `services/creativeWorkshop.ts`

**发布流程**（第346-367行）：
- 先通过 `isOfficialCreativeWorkshopDuplicate` 检查是否与官方模块重复
- 需要登录凭据（`构建必需账号载荷`）
- 模块内容通过 POST 发送到 `/api/workshop/modules`
- 返回的 entry 经过 `规范化当前模块` 处理

**下载流程**（第405-415行）：
- 通过 `entry.downloadUrl` 或默认 API 地址下载
- 响应经过 `读取响应JSON` 解析
- 通过 `规范化当前模块` 规范化

**注入流程**：
- 模块的 `contentBlocks` 通过 `injectionTarget` 字段指定注入目标
- 允许的注入目标：`manualWorldPrompt`, `worldExtraRequirement`, `manualRealmPrompt`, `openingExtraRequirement`, `imageWorkflow`, `referenceOnly`

**发现的问题**：

| 问题 | 严重性 | 位置 | 描述 |
|------|--------|------|------|
| **缺少内容沙箱** | 中 | `creativeWorkshop.ts:269` | `injectionTarget` 允许的值中包含 `imageWorkflow`，模块内容可以直接注入到 AI prompt 中。云端模块的内容未经过滤直接拼入 prompt，存在 prompt 注入风险 |
| **sha256 未校验** | 中 | `creativeWorkshop.ts:280, 405-415` | 下载模块后没有校验 sha256 哈希值，理论上存在中间人篡改风险 |
| **payload 无类型约束** | 低 | `creativeWorkshop.ts:248` | `payload` 类型为 `Record<string, unknown>`，接受任意结构，缺少深度校验 |
| **旧版迁移注入风险** | 中 | `creativeWorkshop.ts:129-226` | `升级旧版开局模块` 直接将旧内容映射到 `manualWorldPrompt`，如果旧数据被污染，会直接影响 AI 行为 |

### 1.2 模块内容对 AI Prompt 的影响

模块通过以下方式影响 AI prompt：
- `manualWorldPrompt` → 注入手动世界观提示词
- `worldExtraRequirement` → 追加到世界观细化要求
- `manualRealmPrompt` → 注入能力/境界提示词
- `openingExtraRequirement` → 追加到开局额外要求

**Prompt 注入风险评估**：**中等**

云端模块的内容字段（`content`）是纯文本，直接拼接进 AI 的 system prompt 或 user prompt。如果恶意模块包含类似 "忽略之前所有指令" 的文本，理论上可以影响 AI 输出。不过：
1. 模块发布需要账号认证
2. 模块内容作为数据而非指令拼接（取决于 prompt 构建方式）
3. 没有发现对模块内容的 prompt 注入防护（如分隔符、标记化）

## 二、小说分解系统

### 2.1 整体架构

**数据流**：
```
原始文本 → 章节提取 → 分段 → AI解析 → 注入树 → 快照 → AI Prompt
```

**关键文件**：
- `novelDecompositionPipeline.ts` — 核心管道
- `novelDecompositionStore.ts` — IndexedDB 持久化
- `novelDecompositionRuntime.ts` — 运行时执行器
- `novelDecompositionScheduler.ts` — 后台调度
- `novelDecompositionInjection.ts` — 注入逻辑
- `novelDecompositionCalibration.ts` — 时间校准
- `novelDecompositionTimelineConstraints.ts` — 跨世界时间线约束

### 2.2 数据完整性保障

**优点**：

1. **严格的输入规范化**：`novelDecompositionStore.ts` 中每个字段都有对应的标准化函数（第285-391行），所有数据在入库前经过深度规范化

2. **时间线连续性校验**：`novelDecompositionPipeline.ts:816-861` 的 `校验分段时间连续性` 确保：
   - 每段有有效的起止时间
   - 下一段起点不早于上一段终点
   - 格式符合 `YYYY:MM:DD:HH:MM`

3. **AI 输出完整性校验**：`校验分段AI输出完整性`（第742-814行）检查：
   - 组号一致性
   - 时间线起止存在
   - 关键事件时间字段完整且逻辑合理
   - 信息可见性标注完整

4. **时间锚点系统**：`novelDecompositionTime.ts` 实现了完整的时间解析、规范化、平移运算，支持中文数字转换

5. **自动重试机制**：`novelDecompositionRuntime.ts:396-466` 支持配置自动重试次数，失败后可重试

**发现的问题**：

| 问题 | 严重性 | 位置 | 描述 |
|------|--------|------|------|
| **原始文本无大小限制** | 中 | `novelDecompositionStore.ts:397` | `原始文本` 字段无长度上限，超大文本可能导致 IndexedDB 写入失败或性能问题 |
| **AI 输出截断无检测** | 中 | `novelDecompositionPipeline.ts:881-897` | 流式输出可能被截断，但 `解析小说拆分分段` 不检测 JSON 不完整的情况 |
| **并发写入风险** | 低 | `novelDecompositionRuntime.ts:393` | 分段处理循环中每次完成一个分段就写入一次 IndexedDB，如果用户在处理中刷新页面，可能产生不一致状态 |
| **分享导入 ID 重映射** | 低 | `novelDecompositionStore.ts:1118-1131` | 导入分享数据时生成新 ID，但如果 `关联分段ID列表` 引用了旧 ID，重映射依赖 `segmentIdMap`，需要确保所有引用都被更新 |

### 2.3 小说分解→重组→注入流程

**流程合理性评估**：**良好**

分解流程设计合理：
1. 章节识别支持多种格式（标题行、段落分割）
2. 分段支持单章/多章/自定义范围
3. AI 解析时有完整的前后组参考
4. 注入树按目标链路分类（main_story/planning/world_evolution）
5. 滑窗机制只注入当前相关章节，控制 prompt 长度

**注入流程**（`novelDecompositionInjection.ts`）：
- 三种注入目标各有不同的内容裁剪策略
- 支持 `maxChars` 上限控制
- 同人角色替换在注入前应用

## 三、世界书系统

### 3.1 世界书匹配/注入逻辑

世界书通过 `creativeWorkshopModules.ts` 中的 `modeWorldbooks` 字段注入。

**关键发现**：

| 问题 | 严重性 | 位置 | 描述 |
|------|--------|------|------|
| **关键词冲突处理缺失** | 中 | `creativeWorkshopModules.ts:74` | 世界书条目的 `关键词` 数组为空 `[]`，依赖 `注入模式: 'always'` 无条件注入。没有关键词冲突检测机制 |
| **优先级覆盖不透明** | 低 | `creativeWorkshopModules.ts:76` | 多个模块的世界书条目优先级可能相同（如模式包中多个条目优先级 90-106），后加载的会覆盖先加载的，但没有冲突报告 |
| **作用域过宽** | 中 | `creativeWorkshopModules.ts:57` | `全流程模式世界书作用域` 包含所有链路（main/opening/world_evolution/variable_calibration/story_plan/heroine_plan/tavern），一个题材口径条目会影响所有 AI 决策链路 |

### 3.2 模块世界书的注入目标

模块的 `contentBlocks` 通过 `injectionTarget` 指定注入位置：
- `manualWorldPrompt` → 手动世界观提示词
- `worldExtraRequirement` → 世界观细化要求
- `manualRealmPrompt` → 手动境界提示词
- `openingExtraRequirement` → 开局额外要求
- `imageWorkflow` → 图像工作流
- `referenceOnly` → 仅参考（不注入）

**安全评估**：`referenceOnly` 是一个好的设计，允许模块包含但不注入敏感内容。

## 四、数据校验

### 4.1 校验充分性评估

| 维度 | 评估 | 说明 |
|------|------|------|
| 类型校验 | ✅ 充分 | 所有数据通过 TypeScript 类型 + 运行时标准化函数 |
| 范围校验 | ✅ 充分 | 数字字段有 min/max 限制，字符串有 trim 和空值检查 |
| 时间格式 | ✅ 充分 | 严格的正则 `^\d{4,6}:\d{2}:\d{2}:\d{2}:\d{2}$` + 逻辑校验 |
| AI 输出结构 | ✅ 充分 | 组号、时间字段、可见性标注都有校验 |
| 内容安全 | ⚠️ 不足 | 无 prompt 注入检测，无敏感内容过滤 |
| 大小限制 | ⚠️ 不足 | 原始文本、章节列表无上限 |
| 关系完整性 | ✅ 充分 | 导入时 ID 重映射，关联列表更新 |

### 4.2 具体校验缺陷

1. **`creativeWorkshopModules.ts`**：模块的 `content` 字段无长度限制，理论上可以注入无限长的 prompt
2. **`novelDecompositionPipeline.ts:421-448`**: 原文内容直接传递给 AI，无内容过滤
3. **`novelDecompositionInjection.ts:325-332`**: `提取命名区块内容` 使用正则匹配，如果区块内容包含 `【】` 字符可能导致匹配错误

## 五、错误处理

### 5.1 网络失败恢复

**优点**：
- `novelDecompositionRuntime.ts:396-466`：自动重试机制，可配置重试次数
- `novelDecompositionRuntime.ts:486-524`：中断处理（暂停/取消）保留已完成进度
- `novelDecompositionStore.ts:743-749`：后台续跑任务筛选

**问题**：

| 问题 | 严重性 | 位置 | 描述 |
|------|--------|------|------|
| **无退避策略** | 中 | `novelDecompositionRuntime.ts:457-464` | 重试之间没有退避延迟，连续失败时可能触发速率限制 |
| **网络错误分类不足** | 低 | `creativeWorkshop.ts:333` | 网络失败统一 `catch` 静默处理，不区分超时/断网/服务端错误 |
| **部分完成状态** | 低 | `novelDecompositionRuntime.ts:508-513` | 分段失败后标记为"失败"但不记录失败时的部分输出 |

### 5.2 格式错误恢复

- AI 返回非预期格式时，`校验分段AI输出完整性` 会抛出错误
- 错误信息包含具体字段名和期望值
- 但没有"尽力解析"的降级策略——如果 AI 少输出一个可选字段，整个分段失败

## 六、安全分析

### 6.1 外部内容隔离

**现状**：❌ **无沙箱机制**

外部内容（云端模块、分享导入的小说数据）的处理方式：
1. 下载后直接使用 `规范化当前模块` 处理类型，不隔离
2. 模块内容直接拼接进 AI prompt
3. 小说原文直接发送给 AI 服务

**风险矩阵**：

| 威胁 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| Prompt 注入（模块） | 中 | 高 | 需要账号认证发布 |
| Prompt 注入（小说） | 高 | 中 | 无防护，但小说内容是用户自己的数据 |
| 恶意分享包 | 中 | 中 | 有格式校验，无内容安全校验 |
| 数据泄露 | 低 | 低 | 模块发布时携带用户凭据 |

### 6.2 凭据处理

**问题**：`creativeWorkshop.ts:62-78` 中 `构建账号载荷` 将用户名和密码明文放入请求体。虽然通过 HTTPS 传输，但：
- 日志中可能意外记录
- 错误堆栈中可能包含

### 6.3 去重机制

`creativeWorkshopDedupe.ts` 使用 JSON 序列化 + 排序键的方式生成指纹，去重逻辑正确但：
- 仅基于内容指纹，不考虑来源信任级别
- 官方模块优先级高于云端模块（通过 `整合创意工坊模式包` 顺序决定）
