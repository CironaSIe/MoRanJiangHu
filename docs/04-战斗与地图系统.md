# 战斗与地图系统审计

## 1. 战斗系统

### 1.1 数据模型完整性

`models/battle.ts` 仅 26 行，极其精简。

| 严重度 | 问题 |
|--------|------|
| ⚠️ 中 | 缺少回合数追踪，`BattleRoundAnimation` 只能硬编码"回合数: 1" |
| ⚠️ 中 | 敌方无部位血量结构，只有单一当前血量 |
| ⚠️ 中 | 缺少战斗日志/历史，动画组件靠 mock 数据 |

### 1.2 战斗思维链 (CoT Combat) — 设计良好

8 步完整思考链：锁定→出手合法性→先机瞄准→接战防御→伤害反馈→态势终局→正文判定行→状态摘要。

**问题**：CoT 与实际数据模型脱节 — CoT 要求输出详细判定值，但 `战斗状态结构` 只存储 `是否战斗中 + 敌方列表`，判定结果不持久化。

### 1.3 战斗结果对角色状态的影响

| 严重度 | 问题 |
|--------|------|
| 🔴 高 | 战斗结束只清敌方，不回写玩家部位血量 |
| 🔴 高 | `enemy.最大血量 <= 0` 判定有逻辑缺陷 |
| ⚠️ 中 | 缺少战斗结果→社交 NPC 死亡的联动 |

### 1.4 数据一致性

| 严重度 | 问题 |
|--------|------|
| 🔴 高 | 两套战斗指标计算公式不一致（BattleModal vs rulebook） |
| ⚠️ 中 | BattleRoundAnimation 使用 mock 数据而非真实战斗状态 |

## 2. 地图系统

### 2.1 六层树结构

```
寰宇 → 大地点 → 中地点 → 小地点 → 区地点 → 子地点
```

| 严重度 | 问题 |
|--------|------|
| ⚠️ 中 | 旧坐标字段未从模型定义中移除（world.ts 地图/建筑等旧字段） |
| ℹ️ 低 | `地点归属结构` 只有三层，缺少区地点/子地点归属 |

### 2.2 坐标系统与渲染 — 整体合理

支持多种坐标输入格式、边界约束、建筑避让、A* 路径规划。

| 严重度 | 问题 |
|--------|------|
| ⚠️ 中 | `读取地图层级类型` fallback 到'小地点'不安全 |
| ⚠️ 中 | `是否室内层级` 关键词过于宽泛 |
| ℹ️ 低 | A* 路径搜索无步数上限 |

### 2.3 地图布局优化

`mapLayoutOptimizer.ts` 与 `mapSpatial.ts` 的 `是否聚落层级` 关键词列表不一致。

## 3. NPC 地图定位

### 设计良好：精确位置与模糊位置分离

| 严重度 | 问题 |
|--------|------|
| 🔴 高 | `地图文本相互命中` 子串匹配有误伤风险（"后山"匹配到"后山小院"） |
| ⚠️ 中 | NPC 无显式位置且 `是否在场=true` 时直接放到当前叶子节点 |
| ⚠️ 中 | RegionMap 中 `npcAtLocation` 对 `isRoom` 的处理有边界问题 |

---

## 4. 地图更新工作流

### 安全设计较好

严格白名单（只接受 push/set/delete 命令）、键名限制、层级规范化、重复名称检测。

| 严重度 | 问题 |
|--------|------|
| 🔴 高 | 对 `set` 命令没有值校验，AI 可清空整个地图 |
| 🔴 高 | 不验证树的连通性（孤儿节点风险） |
| ⚠️ 中 | `memory_regenerate` 模式的 JSON 解析容错不足 |
| ⚠️ 中 | 开局空地图检测条件过于宽松 |

---

## 5. 战斗→地图联动

**战斗和地图系统之间几乎没有直接联动**：

| 严重度 | 问题 |
|--------|------|
| 🔴 高 | 战斗结果不更新地图状态 |
| 🔴 高 | 地图更新工作流不感知战斗事件 |
| ⚠️ 中 | NPC 战斗死亡不会更新地图上的 NPC 标记 |
| ⚠️ 中 | 战斗地点与 `环境.具体地点` 无绑定 |
