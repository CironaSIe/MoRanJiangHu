# 墨色江湖（MoRanJiangHu）系统架构与机制总览

## 1. 项目概述

AI 驱动的文字角色扮演游戏引擎，支持 8 种题材模式（武侠/仙侠/西方奇幻/灵气复苏/都市修仙/现代都市/末日丧尸/无限流），运行在 Web 端（React + TypeScript + Vite），可通过 Capacitor 打包为 Android APK。

---

## 2. 技术栈

| 层级 | 技术 | 用途 |
|------|------|------|
| 前端框架 | React 19.2 + TypeScript 5.8 | UI 渲染与类型安全 |
| 构建工具 | Vite 6.2 | 开发服务器 + 生产构建 |
| 样式方案 | Tailwind CSS 3.4 | 原子化 CSS，day/dark 双主题 |
| 状态管理 | React Hooks（useState/useRef/useCallback） | 无第三方状态库，自建超大 Hook |
| 持久化 | IndexedDB（自定义封装 dbService） | 存档、设置、图片资源的本地存储 |
| AI 接入 | OpenAI Chat Completions 兼容协议 | 支持 Gemini/Claude/DeepSeek/GLM/MiMo 等 |
| 流式输出 | SSE / Capacitor Native Chat Streamer | Web 端 fetch streaming + 安卓原生桥接 |
| 安卓打包 | Capacitor 8 + Gradle | APK 构建 |
| 边缘计算 | Cloudflare Workers | API 路由、APK 分发、图片代理 |
| 存储后端 | OneDrive（OpenList 代理） | 存档/APK/图片的云端存储 |
| 安全 | DOMPurify | HTML 清洗 |
| Token 计算 | js-tiktoken | Token 估算 |

### 4.2 数据更新机制

所有状态更新通过 `stateHelpers.ts` 的命令引擎执行：
- `set` — 赋值（如 `set 角色.当前血量 = 80`）
- `add` — 数组追加（如 `add 社交[0].记忆 "战斗记录"`）
- `push` — 同 add
- `delete` — 删除（如 `delete 社交[0]`）
- `sub` — 数组移除

命令由 `responseCommandProcessor.ts` 解析 `tavern_commands` 后通过 `applyStateCommand()` 执行。

## 5. 核心工作流——一局游戏的完整生命周期

### 5.1 开局阶段

```
用户进入首页 → 点击"新局"
    ↓
NewGame 向导（选择题材模式/预设/自定义）
    ↓
worldGenerationWorkflow.ts → 调用 AI 生成世界观
    ↓
openingStoryWorkflow.ts → 生成开场剧情
    ↓
创建初始状态（角色/环境/世界/社交/记忆/剧情/门派/任务）
    ↓
saveCoordinator.ts → 自动存档 #1
    ↓
进入主游戏界面（view === 'game'）
```

### 5.2 主循环（每回合）

```
用户输入文本 → InputArea
    ↓
sendWorkflow.ts: 执行主剧情发送工作流()
    │
    ├─ [阶段0] 剧情回忆检索（可选，达到触发回合后）
    │    → recallWorkflow.ts → 向量/关键词检索回忆档案
    │    → 生成 <回忆> 标签注入主剧情上下文
    │
    ├─ [阶段1] 构建系统提示词
    │    → systemPromptBuilder.ts → 组装有序消息链
    │    → 包含：角色状态/环境/NPC/世界/剧情/记忆/世界观/难度/...
    │
    ├─ [阶段2] 主剧情 AI 请求
    │    → mainStoryRequest.ts → 构建消息数组
    │    → storyTasks.ts → generateStoryResponse()
    │    → chatCompletionClient.ts → 流式/非流式请求
    │    → storyResponseParser.ts → 解析标签协议
    │    → 校验：标签完整性/人称一致性/NPC姓名/名器泄露/字数
    │
    ├─ [阶段3] 命令执行（立即生效）
    │    → responseCommandProcessor.ts → 解析 tavern_commands
    │    → stateHelpers.ts → applyStateCommand() 更新游戏状态
    │    → 写入记忆（即时记忆 + 短期记忆）
    │    → 更新历史记录
    │
    ├─ [阶段4] 后台队列（异步并行）
    │    ├─ 文章优化（可选，polish）
    │    ├─ 变量生成（variable）—— 独立 AI 调用精化状态变量
    │    ├─ 动态世界（world）—— 独立 AI 调用推演世界变化
    │    ├─ 规划分析（planning）—— 独立 AI 调用修订剧情规划
    │    └─ 地图更新（map）—— 独立 AI 调用增量更新地图层级
    │
    └─ [阶段5] 收尾
         → 回合结束自动存档（可选）
         → 触发 NPC/场景自动生图
         → 后台同步到云端（OneDrive）
```

### 5.3 存读档

```
存档流程：
  saveCoordinator.ts
  → 深拷贝当前完整状态（角色/环境/社交/世界/战斗/剧情/记忆/...）
  → 收集图床图片引用
  → dbService.ts → IndexedDB put()
  → saveArchiveService.ts → 可选导出 ZIP（含 JSON + 图片）
  → cloudPlayService.ts → 后台同步到 OneDrive

读档流程：
  SaveLoadModal.tsx → 选择存档
  → dbService.ts → IndexedDB get()
  → saveLoadWorkflow.ts → 恢复状态到 useGameState
  → 重建 UI
```

## 6. AI 集成机制

### 6.1 调用方式

```
前端（Web）:
  chatCompletionClient.ts
  → fetch('/v1/chat/completions', { stream: true })
  → 解析 SSE event-stream
  → onDelta 回调更新 UI

前端（Android APK）:
  → Capacitor Native Chat Streamer 插件
  → 原生层处理 HTTPS 流式请求（绕过 CORS/代理限制）
  → WebSocket 事件回传 JS

Cloudflare Workers:
  → 部分 API 请求通过 Worker 代理（鉴权/缓存/B2 授权）
```

### 6.2 多模型兼容

| 供应商 | 协议 | 特殊处理 |
|--------|------|----------|
| Gemini | OpenAI-compatible | 默认行为 |
| Claude | OpenAI-compatible | 默认行为 |
| DeepSeek | OpenAI-compatible | Prefix/Prefill 锁格式、接管摘要、Thinking |
| 智谱 GLM | OpenAI-compatible | HTML 注释思维链、锁格式 |
| 小米 MiMo | OpenAI/Anthropic-compatible | 稳定输出预设 |

支持独立模型路由：主剧情/变量/润色/规划可各用不同模型。

## 7. 游戏机制详解

### 7.1 题材模式系统

8 种官方题材模式，分为 7 个分组：

| 模式 | 分组 | 核心体验 |
|------|------|----------|
| 武侠 | `wuxia` | 江湖、门派、内力、武学与凡俗尺度成长 |
| 仙侠 | `xianxia` | 宗门、灵根、坊市、秘境与修真境界体系 |
| 西方奇幻 | `western_fantasy` | 王国、公会、骑士、法师、地下城 |
| 灵气复苏 | `urban_xianxia` | 现代都市突然灵气复苏 |
| 都市修仙 | `urban_xianxia` | 现代城市暗处有修行家族 |
| 现代都市 | `modern` | 非古代、低玄或无玄现代生活 |
| 末日丧尸 | `apocalypse` | 感染机制、生存物资、噪音风险、营地政治 |
| 无限流 | `infinite` | 主神空间、恐怖片轮回、奖励点、基因锁 |

每种题材通过 `构建官方模式运行时配置()` 生成 ModeRuntimeProfile，涵盖：identity/economy/time/organization/ability/items/map/task/npc/image/opening/validation。

### 7.2 战斗系统

- 战斗判定通过 **7 步思维链（CoT Combat）** 完成
- BattleModal.tsx 渲染战场单位系统
- ⚠️ 战斗数据模型过简（battle.ts 仅26行），CoT 判定结果不持久化到状态

### 7.3 NPC 社交系统

- NPC结构（287行定义）：修行/战斗/装备/私密档案/关系网/记忆
- 入档条件：关键作用或累计互动超 3 次
- 关系演化：通过关系突破条件和好感度突破条件推进
- 保护机制：npcRetentionGuard（删除保护）+ npcDeathGuard（死亡保护）

### 7.4 记忆系统

五层架构：即时→短期→中期→长期+回忆档案

- 溢出自动触发 AI 压缩
- 支持手动触发和回忆检索（向量/关键词）
- 回忆检索：recallWorkflow.ts → 预筛本地候选 → AI 深度检索 → 构建标签注入

### 7.5 地图系统

六层树：寰宇→大地点→中地点→小地点→区地点→子地点

- 独立地图更新队列（后故事阶段）
- NPC 精确位置匹配（mapNpcLocation.ts）
- 支持"回忆解析"模式重建地图
- 地图布局优化（mapLayoutOptimizer.ts）：城镇层≥4建筑时自动生成道路网格

### 7.6 物品与装备系统

- 10种物品类型，6种品质等级
- 13个装备槽位
- 负重系统（基础 + 储物袋/戒指扩容）

### 7.7 功法与修炼系统

- 10种功法类型
- 境界体系采用"固定头部+动态区块替换"
- 不同题材有不同境界划分（武侠/仙侠/西幻/末日/无限流）

### 7.8 门派系统

- 详细门派结构：资源/成员/任务/商店/藏经阁
- 职位等级（1-8级）
- 势力系统：实力等级/关系网/互动事件

### 7.9 创意工坊系统

- 5种模块类型：题材/世界规则/开局预设/能力体系/ComfyUI
- 世界书注入机制：contentBlocks → injectionTarget → AI prompt
- 云端模块发布/下载（需账号认证）

### 7.10 任务与约定系统

- 6种任务类型（主线/支线/门派/奇遇/悬赏/传闻），4种状态
- 5种约定性质（情感/交易/赌约/复仇/承诺），5种状态
- ⚠️ 约定状态转换完全依赖AI，程序层无强制校验

## 9. 架构特点总结

1. **超大单体 Hook 架构**：核心逻辑集中在 `useGame.ts`（4171行）和 `sendWorkflow.ts`（2870行）中，通过 props drilling 传递状态和回调
2. **AI 驱动的游戏引擎**：没有传统游戏循环，每回合完全依赖 LLM 生成叙事 + 状态命令
3. **协议化标签通信**：自定义 XML 标签协议实现 AI 输出的结构化解析，支持重试/修复/校验多层容错
4. **多阶段后台队列**：主剧情完成后，文章优化/变量生成/世界演变/规划分析/地图更新作为独立 AI 任务并行执行
5. **全栈 TypeScript**：前端（React）、边缘（Cloudflare Workers）、构建脚本全部 TypeScript
6. **离线优先 + 云同步**：IndexedDB 本地持久化为主，OneDrive 云端同步为辅，支持 GitHub/WebDAV 多通道备份
7. **多供应商 AI 兼容**：通过统一配置层抽象 Gemini/Claude/DeepSeek/GLM/MiMo 等供应商
