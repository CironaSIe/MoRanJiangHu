# 在线功能与创意工坊机制分析

## 一、在线状态机制（onlinePresence.ts）

### 1.1 架构概览

在线状态系统采用 **WebSocket 优先 + HTTP 心跳兜底** 的双通道设计，用于统计当前在线玩家数、维护会话活跃度，并上报轻量级诊断信息。

### 1.2 会话管理

| 常量 | 值 | 说明 |
|------|------|------|
| `HEARTBEAT_INTERVAL_MS` | 60,000ms | HTTP 心跳间隔 |
| `WS_HEARTBEAT_INTERVAL_MS` | 45,000ms | WebSocket ping 间隔 |
| `WS_RECONNECT_MS` | 10,000ms | WebSocket 断线重连延迟 |
| `SESSION_TTL_MS` | 120,000ms | 会话存活判定 TTL |
| `SESSION_RENEW_GRACE_MS` | 15,000ms | 会话续期宽限期 |

会话 ID 生成规则：`web_{Date.now().hex}_{random_hex_12bytes}`，存储在 `localStorage` 的 `moranjianghu.onlineSessionId` 键。每次心跳同时更新 `moranjianghu.onlineSessionLastSeenAt` 时间戳，用于判断会话是否过期。

### 1.3 WebSocket 连接流程

```
startOnlinePresenceHeartbeat()
  ├── connectSocket()              // 建立 WS 连接
  │   ├── open → sendSocketPayload('hello') + 启动 45s ping 定时器
  │   ├── message → markSessionAlive()
  │   ├── close → scheduleReconnect() (10s 后重连)
  │   └── error → closeSocket() + scheduleReconnect()
  ├── tick()                       // HTTP 心跳兜底（仅当 WS 未连接时执行）
  └── visibilitychange → tick()    // 页面切回前台时立即触发一次心跳
```

### 1.4 心跳载荷

`buildHeartbeatPayload` 每次构建以下 JSON：

```json
{
  "sessionId": "web_...",
  "path": "当前路径（截断至 240 字符）",
  "referrer": "document.referrer",
  "versionName": "应用版本名",
  "versionCode": "版本号",
  "platform": "capacitor-android | web",
  "userId": "云端会话用户 ID",
  "username": "云端会话用户名",
  "imageStats": {
    "migrationStatus": "获取本地图片图床迁移状态()"
  }
}
```

**关键优化**：心跳载荷仅包含轻量图片迁移状态，不再扫描全部 IndexedDB（修复了此前因全量扫描导致的周期性卡顿问题）。

### 1.5 公共统计接口

`fetchOnlinePresencePublicStats()` 发起 `GET /api/admin/online?public=1`，返回：

- `onlineCount` — 当前在线数
- `totalRecentCount` — 近期活跃总数
- `onlineSessionCount` — 在线会话数
- `loggedInPlayers24h` — 24h 内登录玩家列表（含时间线片段）
- `topPlayer24h` — 24h 最活跃玩家
- `hourlyHistory` — 每小时历史统计

### 1.6 清理机制

返回的取消函数会清理所有定时器、关闭 WebSocket、移除 `visibilitychange` 监听。

## 二、创意工坊（CreativeWorkshopModal.tsx）

### 2.1 组件结构

`CreativeWorkshopModal` 是玩家贡献内容的主入口，支持两种模块类型：

| 类型 | 说明 |
|------|------|
| `topic` | 完整模式包（含模式专属世界书） |
| `comfy_workflow` | ComfyUI 生图工作流 JSON |

### 2.2 数据来源

模块列表由三方合并去重：

```
列出创意工坊模块() =
  创意工坊模块列表 (官方内置, source='builtin')
  + 读取本地创意工坊模块() (localStorage, source='local')
  + GET /api/workshop/modules (云端, source='cloud')
```

筛选标签：`全部 | 官方预设 | 社区贡献 | 本地导入`。

### 2.3 模块发布流程

```
发布模块(module)
  ├── 校验是否与官方预设重复 (isOfficialCreativeWorkshopDuplicate)
  │   └── 重复 → 抛出错误
  ├── 校验 ComfyUI 工作流可生图 (仅 comfy_workflow 类型)
  │   └── 调用 services/ai/comfyWorkflowValidation.ts
  ├── POST /api/workshop/modules
  │   └── body: { module, contributor, officialFingerprints, auth }
  └── 刷新本地列表
```

**认证**：必须登录联机账号（`cloudPlayService.ts` 的云端会话），`username + password` 随请求发送。匿名发布仅隐藏署名，仍需账号绑定用于后续管理。

### 2.4 模块导入流程

```
导入JSON文件(event)
  ├── 读取 File → JSON.parse
  ├── 从JSON载荷提取创意工坊模块(payload)
  │   ├── payload[] → 递归展开
  │   ├── payload.modules[] → 递归展开
  │   ├── payload.module → 单模块
  │   └── payload.type && payload.title → 直接解析
  ├── 导入本地创意工坊模块(module)
  │   ├── 规范化当前模块(module, 'local')
  │   ├── 去重后插入 localStorage 头部
  │   └── 最多保留 100 个本地模块
  └── 切换到本地导入分区预览
```

### 2.5 贡献草稿结构（贡献草稿）

完整模式包需填写：

- **基础信息**：标题、副标题、描述、类型、题材模式、标签
- **模式元数据**：货币显示、市场名称、市场行为口径、地图口径、技能建议、预设物品关键词、背景/天赋建议
- **三段核心内容**：题材口径 (`topicBody`)、世界规则 (`worldRulesBody`)、能力体系 (`abilityBody`)
- **世界细节生成**：AI 默认生成 或 贡献者自定义（重要人物/势力/地图层级）
- **运行时配置**：通过 `运行时配置分区列表` 13 个分区可视化编辑（题材身份、经济系统、时间系统、组织系统、能力系统、物品系统、地图系统、任务系统、NPC 系统、世界叙事、生图系统、开局系统、校验系统）
- **货币系统**：支持新版动态货币系统（可视化编辑器 + JSON 编辑）和旧版三层货币系统兼容

### 2.6 模块版本管理

同一模块（相同 `baseModuleId`）支持多版本，按 `createdAt` 排序，最新版本优先展示。发布新版本时自动递增版本号。

### 2.7 本地存储键

| 键 | 用途 |
|------|------|
| `creative_workshop_enabled_modules` | 已启用的模块列表 |
| `creative_workshop_local_modules` | 本地导入的模块 |

## 三、小说分解工作台（NovelDecompositionWorkbenchModal.tsx）

### 3.1 组件结构

`NovelDecompositionWorkbenchModal` 是一个轻量级外壳组件，实际工作逻辑全在 `NovelDecompositionSettings` 组件中。

```
NovelDecompositionWorkbenchModal
  └── NovelDecompositionSettings (4481 行，核心工作台)
```

### 3.2 完整工作流

```
1. 导入原文
   ├── 直接粘贴文本
   ├── TXT 文件导入
   └── EPUB 文件导入 (services/epubImport.ts)

2. 拆章
   └── 从原始文本提取章节(dataset)
       ├── 识别 TXT 章节标题行 (novelStructureHeuristics.ts)
       ├── 过滤疑似目录章节
       └── 兜底：按段落长度（3200 字符）切分

3. 分段
   └── 根据章节生成分段列表(dataset)
       ├── 可配置每批章数（默认按 dataset.每批章数）
       ├── 支持 single_chapter 模式（逐章处理）
       └── 生成 小说拆分分段结构 数组

4. AI 逐段解析
   └── 解析小说拆分分段(params)
       ├── 调用 textAIService.generateNovelDecomposition()
       │   ├── 传入原文、组号、章节范围
       │   ├── 传入前组参考 + 时间线终点 + 下一组标题
       │   └── 支持流式输出 (onStreamDelta)
       ├── 规范化 AI 结果到分段()
       │   ├── 时间锚点规范化 (格式: YYYY:MM:DD:HH:mm)
       │   ├── 信息可见性合并
       │   ├── 角色/势力/地点/物品档案去重合并
       │   └── 关键事件时间窗口计算
       ├── 校验分段 AI 输出完整性()
       │   ├── 组号一致性
       │   ├── 时间线起止存在性
       │   ├── 原著硬约束/可提前铺垫需有信息可见性
       │   └── 关键事件时间逻辑（最早≤开始≤最迟≤结束）
       └── 校验分段时间连续性()
           ├── 本组时间格式合法
           ├── 结束时间 ≥ 起始时间
           └── 本组起始 ≥ 上一组结束（时间线衔接）

5. 聚合数据集
   └── 聚合小说拆分数据集(segments)
       ├── 聚合角色档案（最多 120 个）
       ├── 聚合势力档案（最多 80 个）
       ├── 聚合地图地点档案（最多 120 个）
       ├── 聚合物品档案（最多 120 个）
       └── 提取世界观规则、人物关系、势力关系等

6. 构建注入树
   └── 基于分段构建注入树(dataset)
       ├── 当前阶段概括节点 (planning)
       ├── 原著角色档案节点 (planning, world_evolution)
       ├── 原著势力档案节点 (planning, world_evolution)
       ├── 原著地图地点档案节点 (planning, world_evolution)
       ├── 跨世界时间线硬约束节点 (planning, world_evolution, main_story)
       ├── 原著区间概括节点组
       │   ├── 主剧情分段内容 (main_story)
       │   ├── 规划分段内容 (planning)
       │   ├── 世界演变分段内容 (world_evolution)
       │   ├── 规划时间线事件 (planning)
       │   └── 世界演变时间线事件 (world_evolution)
       └── 所有节点按 order 排序

7. 生成同人模式包
   └── novelDecompositionWorkshopBridge.ts
       ├── 构建小说拆分模式包创意工坊模块()
       └── AI 补全小说模式包配置() (可选)

8. 分享/发布
   ├── 导出分享 ZIP (含 manifest.json, decomposition.json, raw-content.json)
   ├── 导入分享 ZIP
   └── 发布到创意工坊 (发布小说分解创意工坊模块)
```

### 3.3 后台调度

`novelDecompositionScheduler.ts` 提供 `小说拆分后台调度服务`，支持：

- 后台续跑（页面关闭后继续处理）
- 暂停/取消中断（通过 AbortController）
- 进度持久化到 `localStorage`
- 限流队列（防止 API 频率限制）

### 3.4 存储格式

数据集版本为 `9`，分享 ZIP 格式标识为 `wuxia-novel-decomposition-zip`，包含：

| 文件 | 内容 |
|------|------|
| `manifest.json` | 格式标识、版本、导出时间 |
| `decomposition.json` | 完整数据集（含分段、档案、注入树） |
| `raw-content.json` | 原始文本 + 章节列表 |

## 四、云同步机制（objectStorageSync.ts）

### 4.1 架构概览

云同步通过 Cloudflare Worker 代理层访问对象存储，当前使用 **OneDrive via OpenList** 作为存储后端。同步系统支持存档和设置的双向同步。

### 4.2 关键参数

| 常量 | 值 | 说明 |
|------|------|------|
| `OBJECT_STORAGE_ROOT_DIR` | `MoRanJiangHu` | 存储根目录 |
| `OBJECT_STORAGE_SAVES_DIR` | `saves` | 存档目录 |
| `OBJECT_STORAGE_CHUNKS_DIR` | `chunks` | 分片目录 |
| `OBJECT_STORAGE_PROXY_PATH` | `/api/object-storage-proxy` | Worker 代理路径 |
| `OBJECT_STORAGE_FALLBACK_CHUNK_THRESHOLD` | 700KB | 超过此大小启用分片 |
| `OBJECT_STORAGE_CHUNK_BASE64_SIZE` | 768KB | 单个分片大小 |
| `OBJECT_STORAGE_UPLOAD_CONCURRENCY` | 3 | 并发上传数 |

### 4.3 同步流程

```
上传同步
  ├── 读取本地存档列表 (dbService)
  ├── 读取云端清单 (GET manifest.json)
  ├── 差异对比
  │   ├── 新增存档 → 上传
  │   ├── 修改存档（hash 不同）→ 更新
  │   └── 相同 hash → 跳过
  ├── 打包存档为 ZIP (saveArchiveService.ts)
  │   └── 超过 700KB → 分片存储
  ├── 并发上传（最多 3 个同时）
  │   └── PUT /api/object-storage-proxy
  └── 更新云端 manifest.json

下载同步
  ├── 读取云端清单
  ├── 对比本地存档
  ├── 下载缺失/过期的存档包
  │   └── 分片文件 → 合并
  └── 导入 IndexedDB
```

### 4.4 同步键（去重标识）

```
手动存档: manual|{时间戳}|{角色名}
自动存档: auto-node|{系列ID}|{节点ID}|{角色名}
```

### 4.5 元数据结构

每个存档包含丰富的元数据：标题、类型、时间戳、设备类型/标签、应用版本、hash、大小、游戏时间、回合数、系列谱系（parentHash/rootHash/lineageDepth）、开局首回合片段摘要。

## 五、图片图床迁移（imageHostService.ts）

### 5.1 架构概览

图片上传通过 Cloudflare Worker 代理 (`/api/image-host/upload`) 路由到上游图床服务。当前上游为 Telegram Bot API（`DEFAULT_IMAGE_HOST_BASE = https://image1.bacon159.pp.ua`）。

### 5.2 上传流程

```
上传DataUrl到图床(dataUrl, options)
  ├── 校验 data URL 格式
  ├── 优化待上传图片
  │   ├── 估算原始大小
  │   ├── 判断是否超过限制（PC 1.5MB / 移动端 900KB）
  │   ├── 超过限制 → Canvas 压缩为 WebP
  │   │   ├── PC: 最长边 1440px, 质量 0.78
  │   │   └── 移动端: 最长边 1080px, 质量 0.68
  │   └── 移动端仍超限 → 二次压缩至 900px, 质量 0.6
  ├── dataUrl → Blob 转换
  ├── 构建上传 URL
  │   └── /api/image-host/upload?kind=image
  ├── 重试循环（默认 3 次，最多 5 次）
  │   ├── POST FormData { file: blob }
  │   ├── 成功 → 跳出循环
  │   └── 可重试失败 → 退避延迟（1200 * attempt² ms, 最大 12s）
  │       └── 可重试: status 0/408/429/5xx 或 network/timeout 错误
  ├── 解析响应
  │   ├── 提取下载链接（兼容多种响应格式）
  │   ├── 提取文件 ID
  │   └── 提取存储类型 (storage 字段)
  └── 返回 { url, id, size, storage }
```

### 5.3 进度回调

`onProgress` 回调提供以下阶段：

| 阶段 | 说明 |
|------|------|
| `prepare` | 准备上传，显示文件大小 |
| `attempt` | 正在上传，显示第 N/M 次 |
| `retry` | 暂时失败，显示退避等待秒数 |
| `success` | 上传成功，显示耗时 |
| `failed` | 最终失败，显示错误原因 |

### 5.4 诊断日志

所有上传操作通过 `recordDiagnosticLog` 记录，包含：文件名、MIME 类型、原始/压缩后大小、是否移动端、代理 URL、上游状态码、请求 ID 等。失败时额外记录响应片段（截断至 500 字符）。

### 5.5 111666 图床

根据 AGENTS.md 记录，`https://i.111666.best` 作为备用图床：

- 上传：`curl -F image=@file https://i.111666.best/image -H "Auth-Token: $TOKEN"`
- 删除：`curl -XDELETE https://i.111666.best/image/PATH -H "Auth-Token: $TOKEN"`
- 特性：裸直接下载返回 403，需带 `Referer` 头才能加载

### 5.6 OneDrive 集成

预设图片（装备/物品缩略图）存储在 OneDrive，通过 OpenList 代理访问：

- 公开 URL: `https://msjh.bacon159.pp.ua/api/preset-image/{name}.png`
- 缩略图: `thumbs/{name}.webp`
- 上传路径: `/Onedrive/MoRanJiangHu/preset-items/<name>.png`
- 上传方式: `PUT /api/fs/put` (raw body, 不能使用 `/api/fs/form`)
