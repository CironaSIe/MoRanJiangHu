# 任务与门派系统机制分析

## 一、任务系统

### 1.1 任务结构（`models/task.ts`）

```
任务结构 {
    标题: string
    描述: string
    类型: 任务类型        // 主线 | 支线 | 门派 | 奇遇 | 悬赏 | 传闻
    发布人: string
    发布地点: string
    任务世界?: string     // 无限流跨世界任务的世界级名称
    推荐境界: string
    截止时间?: string     // YYYY:MM:DD:HH:MM
    当前状态: 任务状态    // 进行中 | 可提交 | 已完成 | 已失败
    目标列表: 任务目标[]
    奖励描述: string[]    // 描述性奖励文本
    奖励已发放?: boolean  // 防重复结算标记
    剧情暗线?: string     // 给 AI 看的暗线说明
}

任务目标 {
    描述: string
    当前进度: number
    总需进度: number
    完成状态: boolean
}
```

### 1.2 任务类型

六种分类（`任务分类列表`）：

| 类型 | 说明 |
|------|------|
| 主线 | 核心剧情推进，由 AI 根据世界观生成 |
| 支线 | 通用支线任务 |
| 门派 | 门派/组织差遣（武侠/仙侠题材专用） |
| 奇遇 | 偶遇机缘类任务 |
| 悬赏 | 通缉/缉拿类任务 |
| 传闻 | 打听消息类任务 |

**类型归一化**（`utils/taskCompat.ts:91`）：当 AI 返回的类型不在预设集合中时，通过关键词匹配自动归类。优先级：门派 → 悬赏 → 传闻 → 奇遇 → 主线 → 支线（默认）。

### 1.3 任务生成

任务来源有三个渠道：

1. **开局生成**（`hooks/useGame/storyState.ts:787`）：`创建开局主线任务()` 根据题材模式生成一条主线任务。例如：
   - 无限流：主神任务倒计时
   - 末日丧尸：求生本能相关目标
   - 仙侠：问道路引
   - 武侠：江湖因缘

2. **变量生成写入**（`prompts/stats/others.ts`）：每回合 AI 通过 `set 任务列表[i].xxx = ...` 命令新增或更新任务。

3. **门派任务转化**（`hooks/useGame/storyState.ts:737`）：`从门派任务创建通用任务列表()` 将 `玩家门派.任务列表[]` 中的门派任务转化为全局 `任务列表[]` 中的通用任务结构。

### 1.4 任务去重

两层去重机制：

**精确去重**（`hooks/useGame/storyState.ts:769`）：
- 去重键 = `类型 + 标题 + 描述/目标描述`（去除空白字符后拼接）
- 相同键的后续任务被过滤

**语义去重**（`utils/taskCompat.ts:132`）：
- 无限流任务特殊处理：
  - 团队重复生存任务（非主线 + 存活/天亮关键词）被过滤，如果已存在主神主线生存任务
  - 同一生存倒计时任务只保留第一条
- 通用指纹去重：归一化文本后比较（去除标点、统一时间表述等）

### 1.5 自动结算（`utils/taskCompat.ts:103`）

`规范化任务自动结算()` 逻辑：
- 遍历所有目标的 `完成状态`，若 `当前进度 >= 总需进度` 则标记完成
- 若所有目标完成且状态非"已完成/已失败"，自动将状态改为"已完成"

### 1.6 奖励发放（`utils/taskRewards.ts`）

`结算已完成任务奖励()` 触发条件：
- `当前状态 === '已完成'` 且 `奖励已发放 !== true`

奖励解析规则（按顺序匹配）：

| 奖励类型 | 匹配模式 | 效果 |
|----------|----------|------|
| 门派贡献 | `门派贡献/营地贡献/组织信用 +N` | 增加 `玩家门派.玩家贡献`、`累计贡献`、`角色.门派贡献` |
| 旧货币 | `金元宝/元宝/银子/银两/铜钱/奖励点/C级支线剧情 +N` | 按层级增加 `角色.金钱` 对应字段 |
| 动态货币 | 匹配 `runtimeProfile.economy.currencySystem` 中的单位名称 | 增加 `角色.金钱.baseAmount` |
| 属性点 | `可分配属性点 +N` | 增加 `角色.可分配属性点` |
| 技艺熟练度 | `XX技能/技艺熟练度 +N` | 增加对应技艺熟练度（上限100），自动更新等级称号 |
| 物品 | `物品名 xN` | 仅记录到账文本，不直接写入背包（背包必须由 AI 变量命令写入） |

奖励结算后标记 `奖励已发放 = true`，防止重复发放。

```

    门派规模?: string
    弟子总数?: number
    战力分布?: 门派战力分布
    财富评级?: string
    月俸规则?: 门派月俸规则

    藏经阁列表?: 门派藏经[]

```

### 2.2 门派任务

```
门派任务 {
    id: string
    标题: string
    描述: string
    类型: 门派任务类型    // 日常 | 悬赏 | 建设 | 历练
    难度: string
    发布日期: string
    截止日期: string
    刷新日期: string     // 日常=每日, 建设=每旬, 历练=每月
    奖励贡献: number
    奖励资金: number
    奖励物品?: string[]
    当前状态: 门派任务状态  // 可接取 | 进行中 | 已完成 | 已失败 | 已过期
}
```

**开局默认门派任务**（`hooks/useGame/storyState.ts:614`）：
- 根据题材生成 3 条默认任务（日常/建设/历练各一条）
- 无限流：据点加固、物资清点、新人分工复盘
- 末日：营地巡查、营地委托、结伴搜救
- 现代：办公室巡查、客户委托、外勤协作
- 武侠/仙侠：山门外市集巡查、药材短缺委托、门中历练

### 2.3 门派贡献与晋升

**贡献体系**：
- `玩家贡献`：当前可用贡献（兑换商品会消耗）
- `累计贡献`：历史累计（只增不减，用于晋升判定）
- 晋升看累计，兑换看当前

**晋升梯队**（`components/features/Sect/SectModal.tsx:39`）：

| 梯队 | 古风武侠 | 末日营地 | 现代组织 | 无限流 |
|------|----------|----------|----------|--------|
| Lv1 | 杂役弟子(0) | 营地成员(0) | 成员(0) | 新人(0) |
| Lv2 | 外门弟子(100) | 营地骨干(150) | 外勤成员(100) | 正式队员(120) |
| Lv3 | 内门弟子(350) | 营地管理人员(600) | 专业骨干(350) | 资深者(500) |
| Lv4 | 真传弟子(900) | 营地核心管理(1600) | 项目骨干(900) | 副队长(1200) |
| Lv5 | 执事(1600) | 营地统治者(4200) | 协调负责人(1600) | 队长(3000) |
| Lv6 | 长老(3200) | — | 负责人(3200) | — |
| Lv7 | 副掌门(6500) | — | — | — |
| Lv8 | 掌门(12000) | — | — | — |

每个等级提供折扣（5%-30%）和特权（藏经阁调阅、任务权限等）。

**职位判定逻辑**（`SectModal.tsx:287`）：
- 无限流：纯按累计贡献判定
- 其他：取存档职位和贡献职位中等级较高者

### 2.4 门派兑换

`门派商品` 结构：
```
{
    id: string
    物品名称: string
    类型: 武学 | 丹药 | 装备 | 材料 | string
    兑换价格: number    // 贡献点
    库存: number
    要求职位: string
}
```

兑换条件：职位等级 ≥ 要求职位 且 当前贡献 ≥ 折后价格 且 库存 > 0

### 2.5 藏经阁

`门派藏经` 结构：
```
{
    id: string
    名称: string
    类型: 功法 | 身法 | 心法 | 杂学 | string
    品阶: string
    简介: string
    要求职位: string
    要求累计贡献: number
}
```

学习条件：职位等级 ≥ 要求职位 且 累计贡献 ≥ 要求累计贡献

### 2.6 月俸系统

`门派月俸规则`：
```
{
    基础俸禄: number
    贡献系数: number
    规模系数: number
    发放说明: string
}
```

月俸计算公式：`基础俸禄 + floor(累计贡献 × 贡献系数) + floor(弟子总数 × 规模系数)`

每月可领取一次，通过 `上次俸禄月份` 字段判定是否已领取。

### 2.7 题材适配

门派系统通过 `获取组织显示文案()` 函数（`SectModal.tsx:84`）实现多题材语义适配：

| 题材 | 门派语义 | 贡献名 | 商城名 | 藏经阁 |
|------|----------|--------|--------|--------|
| 古风武侠 | 宗门 | 贡献点 | 聚宝阁 | 藏经阁 |
| 仙侠 | 宗门 | 贡献点 | 聚宝阁 | 藏经阁 |
| 末日丧尸 | 营地 | 贡献点 | 物资库 | 资料库 |
| 现代都市 | 组织 | 组织信用 | 资源库 | 资料库 |
| 无限流 | 轮回小队 | 奖励点 | 团队商城 | 能力库 |

## 三、约定系统

### 3.1 约定结构（`models/task.ts:47`）

```
约定结构 {
    对象: string           // 约定对象名称
    头衔?: string
    性质: 约定性质        // 情感 | 交易 | 赌约 | 复仇 | 承诺
    标题: string
    誓言内容: string
    约定地点: string
    约定时间: string       // YYYY:MM:DD:HH:MM
    有效时段: number       // 分钟
    当前状态: 约定状态    // 等待中 | 即将到来 | 已履行 | 已违约 | 已作废
    履行后果: string       // UI 显示用
    违约后果: string       // UI 显示用
    背景故事?: string      // AI 上下文
}
```

### 3.2 约定生成

约定通过 AI 变量命令写入 `约定列表[]`，属于前台状态的一部分（`prompts/runtime/opening.ts:165`）。

开局时可生成初始约定；后续通过每回合变量生成新增。

### 3.3 约定触发

约定没有独立的触发检测逻辑，而是通过以下方式驱动：
- **AI 变量写入**：AI 在变量生成阶段根据剧情判断约定是否被履行/违约，直接更新 `约定列表[i].当前状态`
- **UI 展示**：`AgreementModal.tsx` 展示履行/违约后果，提醒玩家
- **系统提示注入**：约定状态被注入到系统提示中（`hooks/useGame/mainStoryRequest.ts:59`），让 AI 在生成剧情时考虑约定

### 3.4 题材适配

约定的 UI 和文案在不同题材下保持统一结构，但 AI 会根据题材生成不同语义的内容（如武侠的"誓言"对应现代的"合同条款"）。

## 四、拍卖行系统

### 4.1 拍卖行状态（`services/auctionHouse.ts`）

```
拍卖行状态 {
    拍卖品列表: 拍卖品记录[]
    交易记录: Array<拍卖品记录 | 交易记录>
    最近补货时间: number
    行情列表?: 拍卖行情[]
    最近行情时间?: number
}
```

### 4.2 拍卖品记录

```
拍卖品记录 {
    ID: string
    物品: 游戏物品
    卖家名称: string
    卖家ID: string
    起拍价: number
    一口价: number
    当前价格: number
    标价货币: 拍卖货币    // 角色金钱的 key
    状态: 拍卖品状态      // 上架中 | 已成交 | 已下架
    上架时间: number
    过期时间: number
    市场标签: string[]
    来源描述: string
    关联事件?: string

    是否限时热点?: boolean
    购买者名称?: string
    成交时间?: number
}
```

### 4.3 物品投放规则

**当前投放路径**（`App.tsx:1328`）：

```
世界演化 → 势力互动 → 世界.拍卖行待投放物品 → 从势力互动投放拍卖品()
```

旧路径（已从剧情正文提取）已废弃，改为从世界势力互动事件中自然流出。

**投放校验**（`auctionHouse.ts:745`）：
1. 必须存在有效物品（有名称和类型）
2. 物品类型必须是自动入市类型：`武器/防具/饰品/消耗品/材料/秘籍/法宝`
3. 物品必须符合当前题材（如武侠不投放仙侠物品，现代不投放江湖物品）
4. 去重：同名+同类型+同品质的已上架物品不重复投放（可堆叠物品合并数量）

**初始补货**（`auctionHouse.ts:644`）：
- `清理并补货()` 在售数量不足时从默认物品池补货
- 每个题材有独立的默认物品池（4 件基础物品）
- 补货数量上限 4 件，目标在售数量 6 件

### 4.4 价格估算

**基础定价**（`auctionHouse.ts:332`）：

```
市场价值 = floor(物品价值 × 品质倍率 × 行情倍率)
```

品质倍率：`1 + (品质权重 - 1) × 0.08`
- 凡品=1.00, 良品=1.08, 上品=1.16, 极品=1.24, 绝世=1.32, 传说=1.40

行情倍率：根据当前行情列表匹配物品类型（0.9~1.3 之间）

**玩家寄售定价**（`auctionHouse.ts:870`）：
- 起拍价 = floor(定价 × 0.75)
- 一口价 = 定价
- 当前价格 = 起拍价

**事件投放定价**（`auctionHouse.ts:708`）：
- 热点物品：价格 × 1.35
- 普通事件：价格 × 1.08

**牙行收购价**（`AuctionHouseModal.tsx:231`）：
- 收购价 = floor(一口价 × 0.82)

### 4.5 行情系统

行情模板按题材分类，每个题材 4~5 条行情：

| 题材 | 行情示例 | 倍率范围 |
|------|----------|----------|
| 武侠 | 镖道吃紧(装备×1.18)、药市缺货(消耗品×1.22) | 0.92~1.25 |
| 现代 | 同城急单(杂物×1.16)、医疗紧张(消耗品×1.20) | 0.93~1.20 |
| 末日 | 净水紧缺(消耗品×1.28)、燃油管制(材料×1.24) | 0.90~1.28 |
| 无限流 | 恐怖片高发期(消耗品×1.30)、基因锁热潮(材料×1.22) | 0.92~1.30 |
| 都市修行 | 暗市问价(材料×1.22)、防护热卖(装备×1.18) | 1.16~1.22 |

行情刷新规则：
- 强制刷新或距上次刷新 ≥6 小时
- 随机选取 2~3 条行情
- 行情有效期 10~18 小时

### 4.6 交易流程

**购买**：
1. 检查非卖家自己的物品
2. 计算底层货币总价（考虑货币层级倍率）
3. 扣除角色金钱
4. 物品写入角色背包
5. 拍卖品状态变为"已成交"
6. 记录交易日志

**玩家寄售**：
- 创建拍卖品记录，有效期 3 天
- 下回合自动成交

**牙行收购**：
- 卖家以 82% 价格将物品卖给牙行
- 物品从拍卖行移除

**撤回**：
- 卖家撤回寄售，物品返回背包

### 4.7 存储与持久化

- 存储键：`moranjianghu_auction_house_v2:{作用域}`
- 作用域由角色姓名 + 生辰 + 游戏初始时间 + 首条历史时间戳组合生成
- 持久化上限：Web 端 80 件拍品 + 80 条交易记录；原生端 32 + 24
- 存储配额超限时自动瘦身（16 件 + 8 条）

## 五、系统间交互

```
┌─────────────────────────────────────────────────────┐
│                    游戏主循环                         │
│                                                      │
│  AI 生成 → 变量命令 → 写入 任务列表 / 约定列表        │
│                                                      │
│  世界演化 → 势力互动 → 拍卖行待投放物品 → 拍卖行      │
│                                                      │
│  门派任务 → 转化为 → 全局任务列表                     │
│                                                      │
│  任务完成 → 奖励结算 → 贡献/货币/技艺/属性点          │
│                                                      │
│  贡献累积 → 职位晋升 → 兑换折扣/藏经阁权限            │
└─────────────────────────────────────────────────────┘
```

**关键设计原则**：
1. 任务/约定通过 AI 变量命令间接写入，不直接由前端逻辑生成
2. 拍卖行物品从世界势力互动中自然流出，不从剧情正文提取
3. 奖励结算由前端解析奖励描述文本执行，物品奖励需 AI 变量命令补充写入
4. 门派系统是多题材通用的组织框架，通过文案映射适配不同世界观
