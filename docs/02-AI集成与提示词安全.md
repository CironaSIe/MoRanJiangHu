# AI 集成与提示词安全审计

## 1. 提示词注入风险

### 1.1 用户输入直接拼接 — 中风险
**文件**: `hooks/useGame/mainStoryRequest.ts:175-183`
```typescript
const latestUserInputAsModel = 包装繁体任务提示([
    '以下是用户最新输入内容：',
    `<用户输入>${params.sendInput}</用户输入>`
].join('\n'), runtimeGameConfig);
```
用户输入被直接包裹在 `<用户输入>` 标签中放入 assistant 角色消息。如果用户输入包含 `</用户输入>` 等标签闭合序列，可能破坏协议结构。

**补充细节**: 用户输入还通过 `mainStoryRequest.ts:285-294` 以不同角色发送：
- 非 GPT 模式：用户输入作为 `assistant` 角色消息，`user` 消息只是固定文本"开始任务"
- GPT 模式：用户输入作为 `user` 角色消息
```typescript
if (!runtimeGptMode) {
    pushEntry('player_input_as_model', '最新用户输入（模型消息）', '助手', latestUserInputRole, latestUserInputAsModel, ...);
}
pushEntry('start_task', runtimeGptMode ? '本回合用户输入' : '开始任务', '用户', 'user',
    runtimeGptMode ? 包装繁体任务提示(params.sendInput, ...) : 包装繁体任务提示('开始任务', ...));
```
非 GPT 模式下用户输入出现在 assistant 角色中，模型可能无法正确理解语义角色。

### 1.2 玩家自定义提示词注入 — 中风险
**文件**: `hooks/useGame/mainStoryRequest.ts:170-172`
`额外提示词` 是玩家自定义内容，未做转义直接注入 system 消息。恶意提示词可以尝试覆盖系统指令。

### 1.3 酒馆预设正则脚本 XSS — 高风险
**文件**: `hooks/useGame/sendWorkflow.ts:1780-1843`
酒馆预设的正则脚本可以对 AI 输出做任意替换。虽然有沙箱渲染和 HTML 清洗两道处理，但正则脚本来源是外部预设，内容不可控。沙箱渲染模式依赖 iframe 隔离，但同源策略下仍有风险。

### 1.4 世界书注入 — 低风险
世界书内容来自本地存档，注入 system prompt。如果存档被篡改（如通过同步），可注入任意指令。

## 2. 响应解析鲁棒性 — 具体代码级问题

### 2.1 `options` 参数被重新赋值
**文件**: `hooks/useGame/sendWorkflow.ts:1050-1073`
```typescript
if (options) {
    const originalOptions = options;
    options = {  // ← 重新赋值参数
        ...originalOptions,
        onRecallProgress: originalOptions.onRecallProgress ? (...) => ... : undefined,
    };
}
```
`options` 是函数参数，被重新赋值。严格模式下可能导致编译错误或运行时意外行为。

### 2.2 `window.setTimeout` 未做环境检测
**文件**: `hooks/useGame/sendWorkflow.ts:1111`
直接使用 `window.setTimeout`，没有 `typeof window !== 'undefined'` 检查。SSR 或 Node.js 测试环境崩溃。

### 2.3 `parseInt` 误用为 `parseFloat`
**文件**: `services/ai/storyResponseParser.ts:234`
```typescript
return parseInt(match[2], 10);
```
函数 `解析标准时间为天数片段` 返回类型声明为 `{ year: number; month: number; day: number; hour: number; minute: number }`，但 `parseInt` 对于非整数字符串会截断小数。如果输入是 `"01.5"`，`parseInt` 返回 `1` 而不是 `1.5`，且不会报错。
**后果**: 时间格式异常时解析结果静默丢失精度，导致 `计算游戏历程天数` 计算错误。

### 2.2 `break` 导致后续行全部丢弃
**文件**: `services/ai/storyResponseParser.ts:349`
```typescript
if (isNonBodyProtocolHeader) break;
```
在 `清理正文残留协议内容` 中，一旦遇到非正文协议标题行，直接 `break` 跳出循环。该行之后的所有正文内容都会被丢弃。
**后果**: 如果 AI 在正文中间错误插入了一个协议标题行（如 `剧情规划: ...`），则标题行之后的所有正文内容都会被截断丢失。

### 2.3 命令值大小无限制
**文件**: `services/ai/storyResponseParser.ts:1325-1358`
`解析命令值` 函数对值类型做了严格区分（字符串去引号、布尔值、数字、JSON 对象），但没有对命令值的深度/大小做限制。一个超大的 JSON 值可能导致内存问题（受最大输出 Token 限制，风险低）。

### 2.4 `inInitLeakBlock` 状态机的空行处理缺陷
**文件**: `services/ai/storyResponseParser.ts:424`
```typescript
if (inInitLeakBlock) {
    if (!line || /^(?:#{1,6}\s*)?\d+[.、]\s*/.test(line) || ...) {
        removedCount += 1;
        continue;
    }
    if (/^[-*]\s+/.test(line)) {
        removedCount += 1;
        continue;
    }
    inInitLeakBlock = false;
}
```
当 `inInitLeakBlock` 为 true 时，空行被 `continue` 跳过，但 `inInitLeakBlock` 状态没有被重置。空行之后的非初始化行也会被丢弃。
**后果**: 初始化泄露块中的空行之后的正常正文内容可能被错误保留或丢弃。

### 2.4 `openStack.splice(lastIndex)` 缺少删除数量参数 🔴 高
**文件**: `services/ai/storyResponseParser.ts:619`
```typescript
const lastIndex = openStack.lastIndexOf(tag);
if (lastIndex >= 0) {
    openStack.splice(lastIndex);  // ← 缺少第二个参数 1
}
```
`splice(lastIndex)` 从 `lastIndex` 开始删除到数组末尾的所有元素。本意是只删除找到的那个标签，但会删除该位置及其之后的所有元素。
**后果**: 标签嵌套栈被错误修改。如栈中有 `['正文', '命令']`，当前标签是 `'正文'`，`splice` 会把 `'命令'` 也一并移除，导致后续标签闭合检测错误。
**同样问题出现在**: `services/ai/chatCompletionClient.ts:619`

### 2.5 `检测正文引号内换行问题` 可能误报
**文件**: `services/ai/storyResponseParser.ts:1146`
```typescript
const nextNonEmptyIndex = lines.findIndex((item, index) => index > lineIndex && item.trim().length > 0);
if (nextNonEmptyIndex >= 0) {
    const opener = stack[stack.length - 1];
    return `正文第${opener.lineNumber}行的引号内容跨行到了第${nextNonEmptyIndex + 1}行...`;
}
```
当 `stack.length > 0`（有未闭合引号）且当前不是最后一行时，无条件查找下一个非空行。如果当前行本身就是引号内的内容（引号在同一行内换行后又闭合了），这个检测会误报。

### 2.6 `提取标题区块内容` 缩进不一致
**文件**: `services/ai/storyResponseParser.ts:147`
标题行用 `line`（trim 后的），后续行用 `rawLine.trimEnd()`，导致区块内容第一行和其他行缩进不一致。

### 2.7 `guard < 20` 硬编码限制
**文件**: `services/ai/storyResponseParser.ts:403`
```typescript
let guard = 0;
while (line.length > 0 && guard < 20) {
    guard += 1;
    // ...
}
```
如果一行中有超过 20 个 HTML 注释片段，循环会提前退出，剩余注释残留在正文中。

### 2.6 错误消息可能泄露 AI 原始内容
**文件**: `services/ai/storyResponseParser.ts:634`
```typescript
const tailSnippet = tail.slice(Math.max(0, tail.length - 80)).replace(/\s+/g, ' ').trim();
issues.push(`疑似输出在 <${lastOpenTag}> 内被截断...末尾片段：${tailSnippet}`);
```
将 AI 响应的最后 80 个字符直接放入错误消息中。如果 AI 输出包含敏感内容（如 NSFW 判定内容、内部变量值），这些内容会出现在错误消息中。

### 2.8 流式 chunk 解析跨帧拼接问题
**文件**: `services/ai/chatCompletionClient.ts:885-1000`
SSE 帧中包含不完整的 JSON 时，`pendingJsonPayload` 可能跨帧拼接错误。

### 2.9 原生流 `streamChat` 异常处理
**文件**: `services/ai/chatCompletionClient.ts:1169`
`streamChat` 抛出异常时 catch 块调用 `settleReject`，但如果 listener 也收到了 `error` 事件，可能导致 `settleReject` 被调用两次（有 `settled` 标志保护，但两次 reject 的错误信息可能不同）。

### 2.10 `new Function()` 执行 — 安全反模式
**文件**: `services/ai/storyResponseParser.ts:1452`
```typescript
const result = new Function(`return (${trimmed})`)() as number;
```
虽然有正则 `/^[0-9+\-*/ .()]+$/` 白名单校验，但 `new Function()` 本质是 `eval` 变体。正则逻辑被修改或绕过时将导致任意代码执行。

### 2.8 `规范化名器档案名称` 被误用于全文本
**文件**: `hooks/useGame/sendWorkflow.ts:294`
```typescript
const bodyText = 规范化名器档案名称(构建玩家可见正文文本(response));
```
`规范化名器档案名称` 函数签名是 `(value: unknown): string`，内部对字符串做 `trim()` 和 `replace(/[\s\u3000]+/g, '')` — 移除所有空白字符。
**后果**: 正文文本中的所有空白被移除后，`bodyText.includes(name)` 检查可能产生误匹配（如"东方不败"在移除空格后可能匹配到"东方不败东方"中的子串）。

## 3. Token 管理

### 3.1 上下文窗口控制
按回合窗口裁剪历史，而非按 token 数裁剪。长时间游戏后可能出现上下文溢出。

### 3.2 最大输出 Token
**文件**: `services/ai/chatCompletionClient.ts:291-307`
按模型设置输出上限（DeepSeek 8K/65K，其他 128K），有用户自定义覆盖。
**问题**: GLM 模型不在任何条件分支中，返回默认 `128_000`。但 GLM 实际最大输出通常远小于此值（如 8K/16K）。
**后果**: 用户设置的 maxTokens 大于 GLM 实际支持值时，API 可能截断输出或报错。

### 3.3 systemPromptBuilder 中 import 位置不规范
**文件**: `hooks/useGame/systemPromptBuilder.ts:30`
import 语句出现在函数定义之后，打破 ES module import 必须在文件顶部的惯例。

### 3.4 `openingConfig` 引用来源不明确
**文件**: `hooks/useGame/systemPromptBuilder.ts:440`
```typescript
构建角色金钱显示快照(
    金钱原始,
    openingConfig || source?.开局配置 || source?.openingConfig,  // ← openingConfig 未在参数中声明
    role
),
```
`openingConfig` 在当前函数作用域中没有定义，可能来自闭包。重构时移动代码会导致 `ReferenceError`。

### 3.5 上下文完整性校验标记过于宽泛
**文件**: `services/ai/chatCompletionClient.ts:452`
```typescript
const 上下文完整性校验标记检测正则 = /\[CTXCHK:\w+\]/;
```
匹配任何 `[CTXCHK:xxxxxx]` 格式字符串，不验证具体 tag 值。如果 AI 碰巧输出类似格式文本，会被误认为校验通过。

### 3.4 已触发上下文截断警告 Set 永不清理
**文件**: `services/ai/chatCompletionClient.ts:418`
```typescript
const 已触发上下文截断警告 = new Set<string>();
```
模块级 Set，一旦加入永远不会移除。API 端点临时故障后恢复，警告也不会再触发。

## 4. 流式输出处理

### 4.1 空闲超时机制 — 优秀
双重超时：首次响应 90 秒、流式空闲 120 秒。超时时如已有完整协议草稿则接受。

### 4.2 abort 事件监听器的竞态条件
**文件**: `hooks/useGame/sendWorkflow.ts:588`
```typescript
const abortByParent = () => {
    requestController.abort(parentSignal.reason || new DOMException('Aborted', 'AbortError'));
};
parentSignal.addEventListener('abort', abortByParent, { once: true });
```
如果 `parentSignal` 在 `addEventListener` 调用之前就已 aborted，`{ once: true }` 的监听器不会被触发。

### 4.3 resolve/reject 判断条件过于宽泛
**文件**: `hooks/useGame/sendWorkflow.ts:596-601`
```typescript
rejectTimeout = (reason?: any) => {
    if (reason && typeof reason === 'object' && 'response' in reason && 'rawText' in reason) {
        resolve(reason as T);  // ← 当作成功
        return;
    }
    reject(reason);
};
```
判断条件 `'response' in reason && 'rawText' in reason` 过于宽泛——任何包含这两个属性的错误对象都会被当作成功结果。

### 4.4 UTF-8 流末尾可能乱码
**文件**: `services/ai/chatCompletionClient.ts:1025`
```typescript
const chunkText = decoder.decode(value, { stream: true });
```
如果流最后一个 chunk 恰好在多字节字符中间，最终 `decoder.decode()` 可能产生乱码。

### 4.5 GPT 模式下用户输入处理差异
**文件**: `hooks/useGame/mainStoryRequest.ts:285-294`
非 GPT 模式下用户输入作为 assistant 消息，user 消息只是固定的"开始任务"。模型可能无法正确理解用户输入的语义角色。

### 4.6 chunk 日志被静默丢弃
**文件**: `services/ai/chatCompletionClient.ts:1327`
```typescript
if (message.includes('stream chunk')) return;
```
所有包含 `'stream chunk'` 的日志被丢弃。如果 chunk 处理出现问题，没有日志可以排查。

## 5. 多模型兼容

### 5.1 DeepSeek Prefix/Prefill
通过 `prefix: true` 标记注入 assistant 前缀消息，不支持时自动关闭。
**问题**: Prefix 只注入 `<thinking>\n`，没有对应闭合标签。如果模型不自动闭合 thinking 标签，后续解析可能将正文内容误认为 thinking 内容。

### 5.2 GLM HTML 注释思维链
将 `<!-- -->` 思维链转换为标准 `<thinking>` 标签。只转换开头的 HTML 注释，正文中间的 HTML 注释由 `清理正文HTML注释残片` 处理。

### 5.3 Gemini system 消息层次结构被扁平化
**文件**: `services/ai/chatCompletionClient.ts:1354`
```typescript
.map((msg) => {
    if (msg.role === 'system') return `【系统规则】\n${content}`;
    if (msg.role === 'assistant') return `【既有回复】\n${content}`;
    return content;
})
```
所有 system 消息被合并为 `【系统规则】` 前缀文本，丢失了原有的分层结构。

### 5.4 响应格式不支持列表硬编码
**文件**: `services/ai/chatCompletionClient.ts:336`
```typescript
if (lowerUrl.includes('doubao') || lowerUrl.includes('volcengine')) return true;
if (lowerUrl.includes('dashscope') && !lowerModel.includes('qwen-max')) return true;
```
硬编码的不支持 `response_format` 供应商列表。新供应商不会被自动检测。

## 6. 命令执行安全

### 6.1 命令校验链 — 优秀
五层校验：女性姓名黑名单、叙事人称一致性、名器档案名称泄露、既有 NPC 姓名改写、既有 NPC 删除风险。

### 6.2 NPC 保护机制 — 优秀
删除风险检测、死亡风险检测、姓名/功法/势力名称保留。

### 6.3 用户输入角色硬编码为 assistant
**文件**: `hooks/useGame/mainStoryRequest.ts:220`
```typescript
const latestUserInputRole: 有序消息角色 = 'assistant';
```
用户输入被标记为 `assistant` 角色。对于不支持 assistant 续写（prefix mode）的 API，这种消息结构可能导致模型行为异常。

### 6.4 DeepSeek Prefix 内容可能不完整
**文件**: `hooks/useGame/mainStoryRequest.ts:317`
```typescript
pushEntry('deepseek_prefix', 'DeepSeek锁格式Prefix', '助手', 'assistant', '<thinking>\n', { prefix: true });
```
只注入 `<thinking>\n`，没有对应闭合标签。模型不自动闭合 thinking 标签时，后续解析将正文内容误认为 thinking 内容。

### 6.5 `标签块已完整闭合` 正则不支持自闭合
**文件**: `hooks/useGame/sendWorkflow.ts:427`
```typescript
return new RegExp(`<\\s*${escaped}\\s*>[\\s\\S]*?<\\s*\\/\\s*${escaped}\\s*>`, 'i').test(text);
```
`[\s\S]*?` 非贪婪匹配对大文本效率较低。
