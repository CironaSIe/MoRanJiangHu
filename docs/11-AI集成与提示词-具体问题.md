# AI 集成与提示词系统——具体代码级问题

## 1. storyResponseParser.ts — 响应解析器

### 问题 1.1：`parseInt` 误用导致时间解析精度丢失
**文件**: `services/ai/storyResponseParser.ts:234`
```typescript
return parseInt(match[2], 10);
```
函数 `解析标准时间为天数片段` 返回类型声明为 `{ year: number; month: number; day: number; hour: number; minute: number }`，但 `parseInt` 对于非整数字符串会截断小数。如 `"01.5"` → `1`，不报错。
**后果**: 时间格式异常时解析结果静默丢失精度，导致 `计算游戏历程天数` 计算错误。

---

### 问题 1.2：`break` 导致后续正文全部丢弃
**文件**: `services/ai/storyResponseParser.ts:349`
```typescript
if (isNonBodyProtocolHeader) break;
```
在 `清理正文残留协议内容` 中，一旦遇到非正文协议标题行，直接 `break` 跳出循环。该行之后的所有正文内容都会被丢弃。
**后果**: AI 在正文中间错误插入协议标题行时，后续正文被截断丢失。

---

### 问题 1.3：`inInitLeakBlock` 状态机空行处理缺陷
**文件**: `services/ai/storyResponseParser.ts:424`
```typescript
if (inInitLeakBlock) {
    if (!line || /^(?:#{1,6}\s*)?\d+[.、]\s*/.test(line) || ...) {
        removedCount += 1;
        continue;  // ← 空行被跳过但不重置状态
    }
    if (/^[-*]\s+/.test(line)) {
        removedCount += 1;
        continue;  // ← 列表行被跳过但不重置状态
    }
    inInitLeakBlock = false;
}
```
`inInitLeakBlock` 为 true 时，空行和列表行被 `continue` 跳过，但状态没有被重置。空行之后的正常正文内容也可能被丢弃。

---

### 问题 1.4：`openStack.splice(lastIndex)` 缺少删除数量参数 🔴
**文件**: `services/ai/storyResponseParser.ts:619`
```typescript
const lastIndex = openStack.lastIndexOf(tag);
if (lastIndex >= 0) {
    openStack.splice(lastIndex);  // ← 缺少第二个参数 1
}
```
`splice(lastIndex)` 从 `lastIndex` 开始删除到数组末尾的所有元素。本意是只删除找到的那个标签，但会删除该位置及其之后的所有元素。
**后果**: 栈中有 `['正文', '命令']`，当前标签是 `'正文'` 时，`splice` 把 `'命令'` 也一并移除，后续标签闭合检测错误。
**同样问题**: `services/ai/chatCompletionClient.ts:619`

---

### 问题 1.5：`guard < 20` 硬编码限制
**文件**: `services/ai/storyResponseParser.ts:403`
```typescript
let guard = 0;
while (line.length > 0 && guard < 20) {
    guard += 1;
    // ...
}
```
一行中有超过 20 个 HTML 注释片段时，循环提前退出，剩余注释残留在正文中。

---

### 问题 1.6：错误消息泄露 AI 原始内容
**文件**: `services/ai/storyResponseParser.ts:634`
```typescript
const tailSnippet = tail.slice(Math.max(0, tail.length - 80)).replace(/\s+/g, ' ').trim();
issues.push(`疑似输出在 <${lastOpenTag}> 内被截断...末尾片段：${tailSnippet}`);
```
AI 响应的最后 80 个字符直接放入错误消息。可能包含 NSFW 内容或内部变量值。

---

### 问题 1.7：`new Function()` 安全反模式
**文件**: `services/ai/storyResponseParser.ts:1452`
```typescript
const result = new Function(`return (${trimmed})`)() as number;
```
正则白名单 `/^[0-9+\-*/ .()]+$/` 校验 + `new Function()` 本质是 `eval` 变体。正则被修改或绕过时导致任意代码执行。

## 2. sendWorkflow.ts — 主发送工作流

### 问题 2.1：`options` 参数被重新赋值
**文件**: `hooks/useGame/sendWorkflow.ts:1050-1073`
```typescript
if (options) {
    const originalOptions = options;
    options = {  // ← 重新赋值参数
        ...originalOptions,
        onRecallProgress: originalOptions.onRecallProgress ? (...) => ... : undefined,
    };
}
```
`options` 是函数参数，被 `const` 声明后不能重新赋值。严格模式下可能导致编译错误。

---

### 问题 2.2：`window.setTimeout` 未做环境检测
**文件**: `hooks/useGame/sendWorkflow.ts:1111`
```typescript
const timeoutId = window.setTimeout(() => { ... }, recallTimeoutMs);
```
直接使用 `window.setTimeout`，没有 `typeof window !== 'undefined'` 检查。SSR 或 Node.js 测试环境崩溃。

---

### 问题 2.3：abort 事件监听器竞态条件
**文件**: `hooks/useGame/sendWorkflow.ts:588`
```typescript
parentSignal.addEventListener('abort', abortByParent, { once: true });
```
如果 `parentSignal` 在 `addEventListener` 调用之前就已 aborted，`{ once: true }` 的监听器不会被触发。

---

### 问题 2.4：resolve/reject 判断条件过于宽泛
**文件**: `hooks/useGame/sendWorkflow.ts:596-601`
```typescript
rejectTimeout = (reason?: any) => {
    if (reason && typeof reason === 'object' && 'response' in reason && 'rawText' in reason) {
        resolve(reason as T);  // ← 当作成功
        return;
    }
    reject(reason);
};
```
任何包含 `response` 和 `rawText` 属性的错误对象都会被当作成功结果处理。

---

### 问题 2.5：`规范化名器档案名称` 被误用于全文本
**文件**: `hooks/useGame/sendWorkflow.ts:294`
```typescript
const bodyText = 规范化名器档案名称(构建玩家可见正文文本(response));
```
函数内部做 `replace(/[\s\u3000]+/g, '')` 移除所有空白字符。
**后果**: `"东方不败"` 在移除空格后可能匹配到 `"东方不败东方"` 中的子串，产生误匹配。

---

### 问题 2.6：`parseDetail` 重复赋值
**文件**: `hooks/useGame/sendWorkflow.ts:234-238`
```typescript
const error = new textAIService.StoryResponseParseError(detail, rawText, detail);
(error as any).parseDetail = detail;  // ← 重复赋值
```
构造函数第三个参数已经是 `parseDetail`，又通过 `as any` 重复赋值。

---

### 问题 2.7：`标签块已完整闭合` 正则效率低
**文件**: `hooks/useGame/sendWorkflow.ts:427`
```typescript
return new RegExp(`<\\s*${escaped}\\s*>[\\s\\S]*?<\\s*\\/\\s*${escaped}\\s*>`, 'i').test(text);
```
`[\s\S]*?` 非贪婪匹配对大文本效率较低。

---

## 3. systemPromptBuilder.ts — 系统提示词组装

### 问题 3.1：import 语句位置不规范
**文件**: `hooks/useGame/systemPromptBuilder.ts:30`
```typescript
import { 规范化游戏设置 } from '../../utils/gameSettings';
// ... 若干行代码 ...
const 解析标准时间为天数片段 = (raw?: string): ... => { ... };
import { 规范化游戏设置 } from '../../utils/gameSettings';  // ← import 在函数定义之后
```
ES module import 必须在文件顶部。

---

### 问题 3.2：`openingConfig` 引用来源不明确
**文件**: `hooks/useGame/systemPromptBuilder.ts:440`
```typescript
构建角色金钱显示快照(
    金钱原始,
    openingConfig || source?.开局配置 || source?.openingConfig,  // ← openingConfig 未在参数中声明
    role
),
```
`openingConfig` 在当前函数作用域中没有定义，可能来自闭包/外层作用域。重构时移动代码会导致 `ReferenceError`。

---

### 问题 3.3：正则替换可能误匹配数值字段
**文件**: `hooks/useGame/systemPromptBuilder.ts:164`
```typescript
.replace(/^(\s*)"(\[\d+\])":\s*\d+,?$/gm, '$1"$2"');
```
如果某个字段值恰好是 `"[0]": 0` 格式（如某个数值字段名为 `[0]`），会被误匹配并移除数值部分。

## 4. mainStoryRequest.ts — 主剧情请求

### 问题 4.1：用户输入角色硬编码为 assistant
**文件**: `hooks/useGame/mainStoryRequest.ts:220`
```typescript
const latestUserInputRole: 有序消息角色 = 'assistant';
```
对于不支持 assistant 续写（prefix mode）的 API，这种消息结构可能导致模型行为异常。

---

### 问题 4.2：GPT 模式下用户输入处理差异
**文件**: `hooks/useGame/mainStoryRequest.ts:285-294`
```typescript
if (!runtimeGptMode) {
    pushEntry('player_input_as_model', '最新用户输入（模型消息）', '助手', latestUserInputRole, latestUserInputAsModel, ...);
}
pushEntry('start_task', runtimeGptMode ? '本回合用户输入' : '开始任务', '用户', '用户',
    runtimeGptMode ? 包装繁体任务提示(params.sendInput, ...) : 包装繁体任务提示('开始任务', ...));
```
非 GPT 模式下用户输入出现在 assistant 角色中，user 消息只是固定的"开始任务"。模型可能无法正确理解用户输入的语义角色。

---

### 问题 4.3：DeepSeek Prefix 无闭合标签
**文件**: `hooks/useGame/mainStoryRequest.ts:317`
```typescript
pushEntry('deepseek_prefix', 'DeepSeek锁格式Prefix', '助手', 'assistant', '<thinking>\n', { prefix: true });
```
只注入 `<thinking>\n`，没有对应闭合标签。模型不自动闭合时，后续正文被当作 thinking 内容处理。

## 5. chatCompletionClient.ts — AI 调用客户端

### 问题 5.1：GLM 模型最大输出上限未被覆盖
**文件**: `services/ai/chatCompletionClient.ts:297`
```typescript
if (model === 'deepseek-chat') return 8_192;
if (model === 'deepseek-reasoner') return 65_536;
if (model.includes('claude') || model.startsWith('gemini-2.5')) return 65_536;
return 128_000;  // ← GLM 走默认值，但实际只支持 8K/16K
```
**后果**: 用户设置 maxTokens 超过 GLM 实际支持值时，API 截断输出或报错。

---

### 问题 5.2：`已触发上下文截断警告` Set 永不清理
**文件**: `services/ai/chatCompletionClient.ts:418`
```typescript
const 已触发上下文截断警告 = new Set<string>();
```
模块级 Set，一旦加入永不移除。API 端点临时故障恢复后，警告不会再触发。

---

### 问题 5.3：校验正则不验证具体 tag 值
**文件**: `services/ai/chatCompletionClient.ts:452`
```typescript
const 上下文完整性校验标记检测正则 = /\[CTXCHK:\w+\]/;
```
匹配任何 `[CTXCHK:xxxxxx]` 格式字符串，不验证具体 tag 值。AI 碰巧输出类似格式文本时误判为校验通过。

---

### 问题 5.4：Gemini system 消息层次结构被扁平化
**文件**: `services/ai/chatCompletionClient.ts:1354`
```typescript
.map((msg) => {
    if (msg.role === 'system') return `【系统规则】\n${content}`;
    if (msg.role === 'assistant') return `【既有回复】\n${content}`;
    return content;
})
```
所有 system 消息合并为 `【系统规则】` 前缀文本，丢失原有分层结构。

---

### 问题 5.5：响应格式不支持列表硬编码
**文件**: `services/ai/chatCompletionClient.ts:336`
```typescript
if (lowerUrl.includes('doubao') || lowerUrl.includes('volcengine')) return true;
if (lowerUrl.includes('dashscope') && !lowerModel.includes('qwen-max')) return true;
```
硬编码列表，新供应商不会被自动检测。

---

### 问题 5.6：UTF-8 流末尾可能乱码
**文件**: `services/ai/chatCompletionClient.ts:1025`
```typescript
const chunkText = decoder.decode(value, { stream: true });
```
流最后一个 chunk 在多字节字符中间时，最终 `decoder.decode()` 可能产生乱码。

---

### 问题 5.7：chunk 日志被静默丢弃
**文件**: `services/ai/chatCompletionClient.ts:1327`
```typescript
if (message.includes('stream chunk')) return;
```
chunk 级别调试信息不可见，排查流式问题困难。
