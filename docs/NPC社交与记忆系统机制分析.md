# NPC社交与记忆系统机制分析

## 一、NPC 数据结构（models/social.ts）

### 1.1 核心接口 `NPC结构`

`NPC结构` 是本系统最复杂的数据实体，包含 **295 行**定义，按职责划分为以下模块：

| 模块 | 关键字段 | 说明 |
|------|----------|------|
| **身份标识** | `id`, `姓名`, `曾用名`, `性别`, `年龄`, `生日` | 唯一 ID + 姓名 + 性别枚举（男/女/男娘/扶她） |
| **修炼体系** | `境界`, `灵根`, `当前灵力/神识`, `丹田状态`, `道基状态`, `心魔值`, `功德`, `业力` | 仙侠题材的核心数值 |
| **社交关系** | `身份`, `关系状态`, `好感度`, `对主角称呼`, `核心性格特征`, `好感度突破条件`, `关系突破条件` | 好感度驱动关系演化 |
| **位置追踪** | `当前位置`, `当前地点`, `位置路径` | 三个字段保持同步，`位置路径` 为完整地点链 |
| **行为意图** | `当前任务`, `行动意图`, `待执行指令`, `指令来源`, `指令时间`, `预期汇合地点` | 玩家可下达指令，NPC 自主执行 |
| **六维属性** | `力量/敏捷/体质/根骨/悟性/福源` | 与主角同口径 |
| **战斗属性** | `攻击力/防御力/当前血量/精力/内力` | 仅队友强制需要 |
| **部位血量** | `头部/胸部/腹部/左手/右手/左腿/右腿` 各 3 字段 | 精确到部位的战斗伤害系统 |
| **装备物品** | `当前装备(NPC装备栏)`, `背包`, `BUFF/DEBUFF`, `技艺[]` | 装备 8 槽位 + 背包 + 状态效果 |
| **外貌描写** | `外貌描写`, `身材描写`, `衣着风格` | 扁平化存储，供生图使用 |
| **私密档案** | `胸部描述`, `小穴描述`, `屁穴描述`, `肉棒描述`, `男娘设定`, `扶她设定` | 按性别选择性填充 |
| **名器系统** | `名器档案[]` | 结构化名器机制，含品质/效果/标签 |
| **亲密边界** | `亲密边界档案` | 矜持度/ASD/欲望/场合敏感度/关系门槛 |
| **孕产系统** | `子宫(子宫档案)` | 含内射记录、生理周期、妊娠档案、生产记录 |
| **初夜/失贞** | `是否处女`, `初夜夺取者`, `失贞档案`, `首次亲密记录[]` | 7 种亲密行为类型的首次记录 |
| **记忆系统** | `记忆(NPC记忆[])`, `总结记忆(NPC总结记忆[])` | 双层 NPC 个人记忆 |
| **图片档案** | `图片档案(NPC图片档案)`, `最近生图结果` | 文生图结果存储 |
| **性转追溯** | `性转记录` | 历史追溯字段 |

## 二、社交关系系统

### 2.1 关系演化机制

关系演化由以下字段驱动：

- **`好感度: number`** — 数值化的好感程度
- **`关系状态: string`** — 当前关系阶段描述（如"陌生人""朋友""恋人"等）
- **`核心性格特征: string`** — 一句话锚定角色主性格，用于关系演化
- **`好感度突破条件: string`** — 下一阶段好感提升的触发条件
- **`关系突破条件: string`** — 关系状态升级/转折的触发条件

## 三、四层记忆系统

### 3.1 全局记忆系统结构（models/system.ts:1206）

```typescript

    短期记忆: string[];         // 短期摘要记忆

    长期记忆: string[];         // 长期精华记忆
}
```

### 3.2 记忆配置结构

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `短期记忆阈值` | 30 | 短期超过此数量触发压缩到中期 |
| `中期记忆阈值` | 50 | 中期超过此数量触发压缩到长期 |
| `重要角色关键记忆条数N` | 20 | 重要角色保留的精确记忆条数 |
| `NPC记忆总结阈值` | 20 | NPC 原始记忆超过此数量触发总结 |
| `即时消息上传条数N` | 10 | 每回合上传给 AI 的即时记忆条数 |
| `剧情回忆检索基础超时秒数` | 25 | 回忆检索基础超时 |
| `剧情回忆检索每10回合追加秒数` | 6 | 每 10 回合额外增加的超时 |
| `剧情回忆检索最大超时秒数` | 180 | 检索超时上限 |

### 3.3 四层记忆的写入与流转（memoryUtils.ts）

**写入流程** (`写入四段记忆`)：

1. **即时记忆**：每回合将 `玩家输入 + AI输出` 合并为一条完整记录，附带游戏内时间戳
2. **短期摘要**：从 AI 的 `shortTerm` 字段提取摘要，附带时间戳
3. **回忆档案**：同步写入结构化回忆条目（含名称/概括/原文/回合/时间戳）
4. **即时→短期溢出**：当即时记忆超过 `immediateLimit`（默认 10），最早条目降级推入短期记忆

**即时与短期合并**：

```
即时记忆条目 + "\n<<SHORT_TERM_SYNC>>\n" + 短期摘要
```

通过分隔符 `<<SHORT_TERM_SYNC>>` 在同一字符串中同时保存详细内容和摘要。

### 3.4 记忆压缩机制（短期→中期→长期）

**自动触发** (`构建待处理记忆压缩任务`)：

```
短期记忆.length > 短期记忆阈值(30) → 压缩前30条 → 中期记忆
中期记忆.length > 中期记忆阈值(50) → 压缩前50条 → 长期记忆
```

**压缩流程**：
1. 取出源层批次（如短期记忆前 30 条）
2. 提取时间范围（从时间戳正则 `【年:月:日:时:分】` 解析）
3. 调用 AI 压缩接口，使用对应提示词模板
4. 生成带时间戳的压缩摘要写入目标层
5. 从源层删除已压缩条目

**手动压缩** (`构建手动记忆压缩任务`)：允许指定来源层和索引范围进行手动压缩。

### 3.5 NPC 个人记忆系统（npcMemorySummary.ts）

每个 NPC 独立维护 `记忆` 和 `总结记忆` 两层：

**总结触发条件**：
- 当 NPC 的原始记忆条数 >= `NPC记忆总结阈值`（默认 20）
- 保留最近 `threshold - 1` 条原始记忆，压缩前面的

**总结索引连续性**：

```typescript
const 取当前原始起始索引 = (summaries) => {
    if (summaries.length === 0) return 0;
    return summaries[summaries.length - 1].结束索引 + 1;
};
```

每次总结的 `开始索引/结束索引` 严格接续上次总结，确保全局索引不断链。

**展示结构**：
- `总结记忆`：带 `[索引范围]` 标签的压缩摘要
- `记忆`：带全局 `[原始索引]` 标签的原始记忆

## 四、剧情回忆检索工作流（recallWorkflow.ts + memoryRecall.ts）

### 4.1 检索流程

```
玩家输入 → 本地预筛候选 → 构建回忆语料库 → AI 检索 → 解析输出 → 构建回忆标签
```

**步骤详解**：

1. **本地预筛** (`预筛剧情回忆候选`)：
   - 对所有回忆档案计算相关度分数
   - 提取检索词（中文 2-3 字切分 + 英文单词）
   - 计算：完整匹配 +12，词项匹配 +1.5~5，近因加成 +0~3
   - 取 topK(24) 高相关 + recentReserve(6) 最近条目

2. **构建语料库** (`构建剧情回忆检索上下文`)：
   - 最近 N 条（默认 20）提供完整原文
   - 更早的条目只提供短期摘要
   - 标记本地预筛结果（"可能相关"）

3. **AI 检索调用**：
   - 使用独立的剧情回忆接口配置
   - 支持流式输出 + COT 思考
   - 输出格式：`强回忆:【回忆001】【回忆005】\n弱回忆:【回忆003】`

4. **回退机制**：AI 失败时，使用本地预筛结果作为回退

5. **构建回忆标签**：
   - 强回忆 → 注入完整原文
   - 弱回忆 → 仅注入摘要概括

### 4.2 回忆档案结构

```typescript
interface 回忆条目结构 {
    名称: string;       // 【回忆001】格式
    概括: string;       // 对应短期摘要
    原文: string;       // 对应即时记忆完整内容
    回合: number;
    记录时间: string;
    时间戳: string;
}
```

## 五、NPC 上下文构建（npcContext.ts）

### 5.1 双轨输出

`构建NPC上下文` 将社交列表分为两个数据块：

| 数据块 | 包含内容 | 详细程度 |
|--------|----------|----------|
| **在场数据块** | `是否在场 === true` 的 NPC | 主要角色用完整数据，其他用基础数据 |
| **离场数据块** | `是否在场 === false` 的 NPC | 含最后互动、离场刷新锚点 |

### 5.2 在场 NPC 数据分级

- **主要角色** (`是否主要角色 === true`)：完整基础数据（含外貌/私密/名器/子宫/记忆）
- **普通在场 NPC**：仅基础数据（姓名/性别/境界/身份/关系状态/好感度/简介/关系网）
- **队友在场 NPC**：额外附加战斗状态（血量/装备/背包）

### 5.3 离场 NPC 刷新锚点

```typescript

}
```

## 六、NPC 生图工作流（npcImageWorkflow.ts）

### 6.1 工作流总览

```
检查开关 → 检查并发锁 → 提取基础数据 → 构建性别约束 → [词组转化器/直出] → 调用生图模型 → 本地化存储 → 写入档案
```

### 6.2 性别硬约束系统

系统对 4 种性别（男/女/男娘/扶她）分别构建了：

- **正向提示词**：确保性别特征正确
  - 女：`1woman, female, adult woman, feminine face, female body`
  - 男：`1man, male, adult man, masculine face, male body`
  - 男娘：`1boy, femboy, passing as female, extremely feminine face...`
  - 扶她：`1woman, futanari, dickgirl, extremely feminine face...`

- **负向提示词**：排除冲突性别特征
- **冲突词清理**：自动移除提示词中与目标性别矛盾的词组
- **年龄处理**：根据年龄生成 `X years old` 提示，修炼长生语境特殊处理

### 6.3 生图控制开关

```typescript
interface 图片功能配置 {
    总开关: boolean;
    NPC开关: boolean;
    使用词组转化器: boolean;
    NPC画风: '二次元' | '国风' | '写实';
    用头像?: boolean;
    用立绘?: boolean;
    用半身?: boolean;
    用私密部位?: boolean;
}
```

### 6.4 并发控制

- `NPC生图进行中集合: Set<string>` — 防止同一 NPC 重复生图
- `NPC生图运行中计数: { current: number }` — 全局限制 1 并发
- 构图去重：已有成功构图的 NPC 不会重复生成同类型构图

### 6.5 提示词构建顺序

```
性别正向词 → 画师串 → 风格正面词 → 角色锚点注入 → 画风附加要求
```

最终提示词 = `构建最终图片提示词(生图词组, 构图, 尺寸, 附加正向, 附加负面, PNG参数)`

## 七、NPC 私密部位生图工作流（npcSecretImageWorkflow.ts）

### 7.1 与普通生图的区别

| 特性 | 普通生图 | 私密部位特写 |
|------|----------|-------------|
| 构图 | 头像/半身/立绘 | 固定 `部位特写` |
| 词组转化器 | 可选用 | 强制使用 NSFW 专用规则 |
| 额外负面词 | 可选 | 默认排除脸/眼/全身/场景等 |
| 裸体语义 | 无 | 可配置强制裸体 |
| 尺寸 | 用户指定 | 默认 `1024x1280` |
| NSFW 模式 | 关闭 | 始终启用 |

### 7.2 NSFW 部位特写规则

词组转化器使用专用提示词模板 `NSFW部位特写词组转化器提示词`，规则包括：
- 目标部位占据绝对主体
- 禁止出现脸、眼睛、头发、手脚、全身
- 必须补齐形状/颜色/肌理/湿润感/体液/光影反射等可画细节
- 单一目标部位、单一镜头、单一画面

### 7.3 部位描述降级

当部位描述为空时，降级使用 `外貌描写 + 身材描写` 作为 fallback，而非直接报错。

## 八、NPC 位置同步机制（responseCommandProcessor.ts）

### 8.1 `同步在场NPC当前位置` 函数

**触发时机**：在每回合命令处理完成后（有命令和无命令两条路径都会执行）

**逻辑**：

```
1. 从环境信息提取当前位置（优先 具体地点，降级到 小/中/大地点）
2. 构建完整位置路径
3. 遍历社交列表中的在场 NPC：
   - 若 NPC 已有明确位置（6 个位置字段任一非空）→ 跳过
   - 否则 → 写入当前位置和位置路径
```

**位置字段检查顺序**：
```
位置路径 → 当前位置 → 当前地点 → 所在地点 → 所在位置 → 具体地点
```

只要任一字段有值就视为"已有明确位置"，不再覆盖。

### 9.3 重建逻辑

1. **清除旧性别专属字段**：删除与新性别矛盾的字段

主角性转同样通过 `处理性转档案重建` 处理，检测 `set 角色.性别 = *` 命令。

## 十一、关键设计决策总结

1. **记忆四层架构**：即时→短期→中期→长期，通过 AI 压缩实现信息逐级提炼，平衡记忆精度与上下文窗口限制

2. **即时/短期同存**：通过 `<<SHORT_TERM_SYNC>>` 分隔符在同一字符串中保存详细内容和摘要，写入时合并、读取时拆分

3. **NPC 个人记忆独立总结**：每个 NPC 维护自己的 `记忆/总结记忆` 双层结构，索引全局不断链

4. **性别硬约束**：生图前构建性别正向/负向提示词 + 冲突词清理 + 强制性别词组，四层防护确保性别一致性

5. **位置同步保守策略**：只填充无位置的在场 NPC，不覆盖已有明确位置的 NPC

6. **性转重建映射表**：通过预定义的字段清除/补齐映射，确保性别转换后档案一致性

7. **并发安全**：全局 Set + 计数器限制 NPC 生图为 1 并发，防止资源争抢
