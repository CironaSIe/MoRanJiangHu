# 战斗与地图系统——具体代码级问题

## 1. models/battle.ts — 战斗模型

### 问题 1.1：`战斗力` 字段类型与实践不一致
**文件**: `models/battle.ts:13`
```typescript
战斗力: number;  // ← 必填字段
```
其他基础属性（力量/敏捷等）全是可选的，唯独战斗力是必填。AI 可能只给粗略数值而不单独计算战斗力。未提供时运行时值为 `undefined`，后续比较产生 `NaN`。

### 问题 1.2：缺少战斗扩展字段
**文件**: `models/battle.ts:23-26`
```typescript
export interface 战斗状态结构 {
    是否战斗中: boolean;
    敌方: 战斗敌方信息[];
}
```
缺少：玩家战斗状态（回合数/buff/debuff）、战斗日志、战斗结果、先手/回合顺序。需要更复杂战斗机制时成为瓶颈。

### 问题 1.3：敌方无部位血量结构
`战斗敌方信息` 只有单一 `当前血量`，没有 `头部/胸部/腹部/手脚` 的部位血量。与角色模型（`character.ts` 有详细部位血量）不对称。

## 3. utils/mapSpatial.ts — 地图空间系统

### 问题 3.1：`稳定散列` 可能产生碰撞
**文件**: `utils/mapSpatial.ts:155-161`
```typescript
const 稳定散列 = (text: string): number => {
    let hash = 0;
    for (let index = 0; index < text.length; index += 1) {
        hash = (hash * 31 + text.charCodeAt(index)) >>> 0;
    }
    return hash;
};
```
简单多项式散列，中文文本碰撞概率不低。`生成地图对象ID` 依赖此散列生成唯一 ID。
**后果**: 不同地点名称产生相同散列时 ID 相同，在 `layerById`/`buildingLookup` 中互相覆盖。

### 问题 3.2：`生成地图对象ID` 空参数时返回固定 ID
**文件**: `utils/mapSpatial.ts:163-171`
```typescript
if (normalizedParts.length === 0) {
    return `${prefix}-0`;  // ← 所有空参数调用都返回相同 ID
}
```
多个调用传入空值时都生成 `layer-0`、`building-0`，ID 冲突。

### 问题 3.3：A* 算法实际是 BFS 🔴
**文件**: `utils/mapSpatial.ts:710-755`
```typescript
const queue = [start];
while (queue.length > 0) {
    const current = queue.shift()!;  // ← FIFO = BFS，不是 A*
    dirs.forEach((dir) => {
        queue.push(next);  // ← 无 g-cost/h-cost 分离
    });
}
```
1. `queue.shift()` 是 FIFO，这是 BFS 不是 A*
2. `dirs` 排序基于固定 `start` 到 `end` 方向，循环中 `current` 变化后排序失效
3. 无 g-cost 和 h-cost 分离
**后果**: 大网格上探索大量不必要的节点，路径效率低。

### 问题 3.4：`nearestFreeCell` 螺旋搜索低效
**文件**: `utils/mapSpatial.ts:695-709`
```typescript
for (let radius = 1; radius <= Math.max(maxX, maxY); radius += 1) {
    for (let dx = -radius; dx <= radius; dx += 1) {
        for (let dy = -radius; dy <= radius; dy += 1) {
```
起点被障碍物包围时需遍历整个网格。64×48 的寰宇层最坏 O(W×H)。

### 问题 3.5：`推出建筑内部点` 可能不稳定
**文件**: `utils/mapSpatial.ts:597-629`
```typescript
layerBuildings.forEach((building) => {
    // 只遍历一次，推出后进入另一个建筑范围不会被再次处理
    if (nearest === 'left') next.x = bounds.minX - gap;
    // ...
});
```
1. 只遍历一次，推出后又进入另一建筑范围不处理
2. 无循环检测，两个建筑相邻时点被推来推去
3. `限制数值` 可能把点推回建筑内部
**后果**: 道路路径点仍落在建筑内部。

### 问题 3.6：`避让层级建筑重叠` 贪心策略可能无法收敛
**文件**: `utils/mapSpatial.ts:560-595`
```typescript
let attempts = 0;
while (placedBounds.some((placed) => 建筑边界重叠(bounds, placed)) && attempts < 36) {
    const columnCount = Math.max(1, Math.floor((layer.网格宽度 - 1.2) / (width + 0.8)));
    // ...
    attempts += 1;  // ← 最大 36 次
}
```
1. 建筑数量多、层级小时 36 次不够
2. `columnCount` 为 1 时所有建筑放同一列，row 递增超出层级高度
**后果**: 建筑仍重叠或推出可视区域。

### 问题 3.7：`是否室内居所位置` 关键词过于宽泛
**文件**: `utils/mapSpatial.ts:338-360`
```typescript
const 是否室内居所位置 = (text): boolean => 文本包含任一(text, [
    '寝居', '卧房', ..., '阁', '堂', '院', '庄', '内部', '室',
]);
```
单独的 `'室'` 字匹配任何包含"室"的文本（"办公室"/"室外"等）。`'阁'`/`'堂'`/`'院'`/`'庄'` 也非常宽泛。
**后果**: 野外层级被误判为室内，使用错误的布局算法。

### 问题 3.8：`计算矩形布局` 建筑可能超出边界
**文件**: `utils/mapSpatial.ts:783-805`
```typescript
const cellWidth = Math.max(4, (layer.网格宽度 - padding * 2) / columns);
const rectWidth = Math.max(4.2, Math.min(8.2, cellWidth * 0.82));
```
total=16 时 columns=4，24×24 层 cellWidth=5，rectWidth=4.2。4 列 × 4.2 + 间距 > 24，超出边界。

## 4. utils/mapNpcLocation.ts — NPC 地图定位

### 问题 4.1：`地图文本相互命中` 子串匹配过于宽松
**文件**: `utils/mapNpcLocation.ts:20-26`
```typescript
export const 地图文本相互命中 = (left, right): boolean => {
    const a = 归一化NPC地图文本(left);
    const b = 归一化NPC地图文本(right);
    if (a === b) return true;
    return (a.length >= 2 && b.includes(a)) || (b.length >= 2 && a.includes(b));
};
```
长度 ≥ 2 的子串匹配过于宽松。"青云"同时匹配"青云山"和"青云镇"。"洛阳"匹配"洛阳城"。
**后果**: 一个在后山的 NPC 被匹配到后山小院。

### 问题 4.2：`选择NPC匹配地图节点` fallback 逻辑错误
**文件**: `utils/mapNpcLocation.ts:140-168`
```typescript
if (!hasExplicitLocation && npc?.是否在场 === true) return currentNode;
```
NPC 无显式位置但 `是否在场=true` 时直接返回当前节点。`是否在场` 由 AI 设置，不一定准确。
**后果**: 不在当前场景的 NPC 被错误显示在当前地图上。

## 5. mapUpdateWorkflow.ts — 地图更新工作流

### 问题 5.1：`解析JSON块` 无 try-catch 🔴
**文件**: `hooks/useGame/mapUpdateWorkflow.ts:256-268`
```typescript
const 解析JSON块 = (rawText: string): any => {
    let text = (rawText || '').trim();
    const thinkEnd = text.lastIndexOf('</思考>');
    if (thinkEnd >= 0) text = text.slice(thinkEnd + '</思考>'.length).trim();
    const firstBrace = text.indexOf('{');
    const lastBrace = text.lastIndexOf('}');
    if (firstBrace >= 0 && lastBrace > firstBrace) { text = text.slice(firstBrace, lastBrace + 1); }
    return JSON.parse(text);  // ← 无 try-catch
};
```
1. thinking 块中讨论了 `{}` 内容时，`firstBrace`/`lastBrace` 可能匹配到错误括号对
2. AI 输出多个 JSON 块时只取第一个 `{` 到最后一个 `}`，合并成无效 JSON
3. `JSON.parse` 失败时直接抛出异常
**后果**: 整个地图更新流程中断。

### 问题 5.2：`add` 动作被静默忽略
**文件**: `hooks/useGame/mapUpdateWorkflow.ts:378`
```typescript
if (action !== 'push' && action !== 'set' && action !== 'delete') return;  // ← add 被忽略
```
正则匹配了 `push|set|add|delete` 四种动作，但实际只处理三种。AI 使用 `add` 添加新地点时命令不生效。

### 问题 5.3：`push` 命令丢失势力字段
**文件**: `hooks/useGame/mapUpdateWorkflow.ts:380-394`
```typescript
result.push({

        // ← 丢失了 控制势力/势力影响/势力标签
    }

```

### 问题 5.4：`构建地图层级替换结果` 父级 ID 解析顺序问题
**文件**: `hooks/useGame/mapUpdateWorkflow.ts:327`
```typescript
父级ID: node.父级ID ? (nameToId.get(node.父级ID) || oldNameToId.get(node.父级ID) || node.父级ID) : '',
```
`normalizedNodes.map(...)` 按数组顺序遍历，子节点在父节点之前出现时 `nameToId` 中还没有父节点 ID。

### 问题 5.5：`构建回忆库地图线索` 可能超出上下文限制
**文件**: `hooks/useGame/mapUpdateWorkflow.ts:75-120`

### 问题 5.6：`规范化层级` 将所有未知层级统一归为 `'区地点'`
