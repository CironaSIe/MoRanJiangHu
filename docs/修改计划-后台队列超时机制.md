# 后台队列超时机制完整修复计划

> 发现核心问题：后台队列的取消/超时机制不完整，导致 AI 端点挂起时整个队列卡死。

---

## 问题分析

### 现状
```
sendWorkflow.ts 后台队列:
  controller (AbortController) ← 只在用户发新消息时 abort
    ↓
  执行可重试独立阶段 (检查 controller.signal.aborted)
    ↓
  deps.执行变量生成并合并响应() ← 无 signal 参数，无法被取消
    ↓
  variableModelWorkflow.ts ← params.signal 来自上游，但上游没有传
    ↓
  chatCompletionClient.ts fetch ← 有 5 分钟兜底超时

问题链：
1. 执行变量生成并合并响应 接口不传 signal → 无法取消
2. 执行地图更新 接口不传 signal → 无法取消
3. 执行可重试独立阶段 没有阶段级超时 → 只能等 fetch 兜底 5 分钟
4. 文章优化、动态世界、规划分析 同理
```

### 设计目标
1. 每个阶段有独立的阶段级超时（变量生成 90s，地图更新 60s，其他 120s）
2. 超时后自动 skip 该阶段，不阻塞后续阶段
3. 不破坏现有的取消机制（用户取消仍然有效）
4. 错误信息通过现有的 onError/onSkip 流程传递

---

## 修复方案

### 第一步：为 `执行变量生成并合并响应` 和 `执行地图更新` 接口添加 signal 参数

**文件**: `hooks/useGame.ts`（接口定义）+ `hooks/useGame/sendWorkflow.ts`（调用处）

```typescript
// 接口新增 signal 参数
执行变量生成并合并响应: (params: {
    // ... 现有参数 ...
    signal?: AbortSignal;  // 新增
}) => Promise<{ ... }>;

执行地图更新: (params: {
    // ... 现有参数 ...
    signal?: AbortSignal;  // 新增
}) => Promise<{ ... }>;
```

### 第二步：在 `sendWorkflow.ts` 后台队列中为每个阶段创建带超时的 signal

```typescript
// 后台队列启动时
const 后台队列Controller = new AbortController();
const 合并信号 = (timeoutMs: number) => {
    const timeoutController = new AbortController();
    const timer = window.setTimeout(() => timeoutController.abort(), timeoutMs);
    // 父 controller 取消时也取消子 controller
    controller.signal.addEventListener('abort', () => timeoutController.abort(), { once: true });
    return { signal: AbortSignal.any([controller.signal, timeoutController.signal]), timer };
};
```

### 第三步：在阶段调用处传入带超时的 signal

```typescript
// 变量生成
const { signal: variableSignal, timer: variableTimer } = 合并信号(90_000);
run: () => deps.执行变量生成并合并响应({
    // ... 现有参数 ...
    signal: variableSignal,
})

// 地图更新
const { signal: mapSignal, timer: mapTimer } = 合并信号(60_000);
run: () => deps.执行地图更新({
    // ... 现有参数 ...
    signal: mapSignal,
})
```

### 第四步：在 `variableModelWorkflow.ts` 和 `mapUpdateWorkflow.ts` 中传递 signal 到 AI 调用

```typescript
// variableModelWorkflow.ts
const 请求变量模型 = (retryHint = '') => textAIService.generateVariableCalibrationUpdate(
    { ... },
    variableApi,
    params.signal,  // 已经传了，确认上游传入即可
    ...
);
```

### 第五步：超时错误处理

超时后 `AbortError` 被 `执行可重试独立阶段` 捕获，走现有的 `onError` → `请求独立阶段失败决策` 流程。需要确保超时错误信息清晰：

```typescript
// 在 合并信号 中
timeoutController.signal.addEventListener('abort', () => {
    reject(new Error(`${stageLabel}请求超时（${timeoutMs/1000} 秒），已跳过继续后续阶段`));
}, { once: true });
```

---

## 超时配置

| 阶段 | 超时 | 理由 |
|------|------|------|
| 文章优化 | 120s | 正文润色可能较长 |
| 变量生成 | 90s | JSON 解析+生成，中等耗时 |
| 动态世界 | 90s | 已有（执行世界演变带超时） |
| 规划分析 | 90s | 已有（执行开局规划带超时） |
| 地图更新 | 60s | 轻量任务，应该很快 |

---

## 验证方式

1. 模拟 AI 端点超时（使用延迟响应的测试 API）
2. 确认超时后阶段被 skip，后续阶段继续执行
3. 确认用户主动取消仍然有效
4. 确认正常响应时超时逻辑不影响流程
