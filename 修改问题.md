## 一、修改计划 ver 6 设计逻辑评估

### 1.1 核心设计评估

**整体判断：设计逻辑基本合理，体现了对元叙事节奏系统的正确理解。**

#### 优势

1. **参数调整合理**
   - 上限从 16 提升到 32：扩大了观测窗口，能更细腻地区分节奏状态
   - 无标签增量从 1 改为 2：明确区分"延续"（+1）和"间歇"（+2）的速度差异
   - 记录从 10 条扩展到 30 条：能追踪更长线的伏笔，避免过早截断

2. **架构边界清晰**
   - 明确定位为"观测系统"而非"控制系统"
   - 不主动操控 AI 行为，只提供描述性上下文
   - 系统、世界演变、剧情规划三者职责分离明确

3. **多标签处理策略**
   - 取最低计数值（最冲击性事件主导）符合叙事直觉
   - 同一回合可以同时有延续和介入，让 AI 有灵活空间

4. **阈值文本设计**
   - 使用"远处""似乎"等距离措辞，避免成为"触发器"
   - 六档分级（12-14, 15-18, 19-22, 23-26, 27-30, 31-32）提供渐进式提示

5. **设计原则正确**
   - "因果潜力"而非"场景类型"判定
   - 四前缀状态覆盖叙事节奏关键转折点
   - LLM 不分关键 NPC/龙套，保持标签纯粹性

## 二、世界事件改造的必要性分析

### 2.1 问题重述

**核心矛盾**：世界演变系统生成的事件缺乏"主角参与度"标记，导致两个问题：

1. **远处事件被强拉到主角身边**
   - 城东发生灭门案（客观事件），主角在西山砍柴
   - AI 可能强行让主角"恰好路过""听到呼救"，破坏世界独立性

2. **主角因果链被系统忽略**
   - 主角救了张三（因果事件），张三应该后续报恩
   - 但世界演变系统不处理"主角参与过的因果链"
   - 若用剧情规划承载，又缺乏时间自动推进机制

**关键认识**：你提到的区分非常准确——

> 世界事件是针对势力、大事件的，从因果上属于**因果很深的类型**。主角相关事件可能被自动遗忘，但世界事件中现有的这堆并不是。

这揭示了**两类事件的本质差异**：

| 事件类型 | 因果强度 | 遗忘机制 | 触发方式 | 示例 |
|---------|---------|---------|---------|------|
| **世界级事件** | 深（势力、大事件） | 不遗忘 | 时间驱动 | 城东灭门案、朝廷换帅、门派内讧 |
| **主角因果事件** | 浅（个人恩怨） | 可遗忘 | 概率触发 | 张三报恩、仇家寻仇、欠债讨债 |

### 2.2 世界事件改造的核心需求

**目标**：让世界演变系统能够区分"远处独立事件"和"主角可能遇到的事件"，避免"主角上山砍柴都被拉进主线"。

#### 改造方案：追加"主角参与度"字段

```typescript
export interface 世界待执行事件结构 {
  事件名: string;
  类型: string;
  事件说明: string;
  计划执行时间: 游戏时间格式;
  // ... 现有字段 ...
  
  // 新增字段
  主角参与度: number;              // 0.0~1.0，标记事件与主角的关联强度
  触发距离?: '远处' | '附近' | '当场';  // 辅助字段，帮助 AI 理解空间关系
  关系强度?: number;               // 若与主角有因果，记录强度（0.0~1.0）
}
```

#### 参与度分级规则

| 参与度范围 | 触发距离 | AI 感知方式 | 强制性 | 示例 |
|-----------|---------|------------|-------|------|
| **0.0 - 0.2** | 远处 | 只能通过"听说""传闻""榜文"获知 | 可选 | 城东昨夜灭门、西域战事、朝廷政变 |
| **0.3 - 0.5** | 附近 | 可能"路过时看见""恰好遇到"，但不强制 | 随机 | 集市纠纷、路遇劫匪、偶遇故人 |
| **0.6 - 0.8** | 当场 | 主角在场或直接相关 | 高概率 | 师门任务、约定事件、追兵到来 |
| **0.9 - 1.0** | 当场 | 主角核心事件，必然触发 | 强制 | 个人生死危机、章节主线、重要镜头 |

#### 系统行为调整

**在 `systemPromptBuilder.ts` 注入世界事件时**：

```typescript
const 构建世界事件上下文 = (世界: 世界数据结构): string => {
  const 远处事件 = 世界.待执行事件.filter(e => e.主角参与度 <= 0.2);
  const 附近事件 = 世界.待执行事件.filter(e => e.主角参与度 > 0.2 && e.主角参与度 <= 0.5);
  const 当场事件 = 世界.待执行事件.filter(e => e.主角参与度 > 0.5);
  
  const 上下文片段: string[] = [];
  
  if (远处事件.length > 0) {
    上下文片段.push('远处传闻：' + 远处事件.map(e => e.事件说明).join('；'));
  }
  
  if (附近事件.length > 0) {
    上下文片段.push('附近动静：' + 附近事件.map(e => e.事件说明).join('；'));
  }
  
  if (当场事件.length > 0) {
    上下文片段.push('当前事件：' + 当场事件.map(e => e.事件说明).join('；'));
  }
  
  return 上下文片段.join('\n');
};
```

**效果**：
- 参与度 0.1 的"城东灭门案"会被注入为"远处传闻：城东武陵山庄昨夜遭灭门"
- AI 只能让主角通过"听说""看到榜文"获知，不能强行让主角路过现场
- 参与度 0.9 的"追兵突袭"会被注入为"当前事件：仇家追兵已逼近"
- AI 必然会让这个事件在本回合或近期回合发生

## 三、因果池机制的定位与边界

### 3.1 因果池 ≠ 世界事件

**核心认识**：世界事件和因果池处理的是**两种不同强度的因果链**。

| 维度 | 世界事件 | 因果池 |
|------|---------|-------|
| **因果强度** | 深（势力级、世界级） | 浅（个人级） |
| **遗忘机制** | 不遗忘 | 可概率遗忘 |
| **触发保证** | 必然发生（时间到期） | 概率触发 |
| **管理者** | 世界演变系统 | 独立因果池系统 |
| **示例** | 朝廷换帅、门派内讧、城东灭门案 | 张三报恩、李四寻仇、欠债讨债 |

**设计原则**：
- 世界事件：因果很深，不能遗忘，必须发生
- 因果池事件：因果较浅，可以遗忘，概率发生（20人中只触发2-3人）

### 3.2 因果池机制的作用

**解决的问题**：批量个人因果链的管理负担。

**场景示例**：
- 主角在战场上救了 20 个伤兵
- 现实逻辑：只有 2-3 人后续能找到主角报恩，其余人可能战死、失联、遗忘恩情
- 当前问题：LLM 需要追踪全部 20 个因果链，认知负载爆炸
- 因果池方案：LLM 只写一次"添加到因果池"，系统自动管理概率触发

**数据结构**：

```typescript
export interface 世界因果池结构 {
  池名称: string;                    // "主角救过的人" "主角得罪的势力"
  池类型: '个人' | '势力' | '物品';
  成员列表: Array<{
    标识: string;                     // 人名/势力名
    关系强度: number;                 // 0.0~1.0，影响触发概率
    记录时间: 游戏时间格式;
    记录回合: number;
    附加信息?: string;
  }>;
  触发策略: '随机抽取' | '全部触发' | '最强者触发';
  每次触发数量: { 最小: number; 最大: number };
  基础触发概率: number;              // 0.0~1.0
  时间衰减系数: number;              // 每回合衰减量
  冷却回合: number;
  上次触发回合?: number;
}
```

### 3.3 因果池与世界事件的协调方式

**流程**：

```
第 5 回合：主角救了张三
  ↓
LLM 在 <剧情规划> 中写入：
  因果池操作: {
    添加成员: [{
      池名称: "主角救过的人",
      成员: {标识: "张三", 关系强度: 0.8}
    }]
  }
  ↓
系统更新 世界数据结构.因果池列表
  ↓
第 15 回合：系统自动检测因果池触发
  - 基础触发概率 0.15 - 时间衰减 0.05 × 10 = 0.10（10%）
  - 随机抽取：20 人中只有 2-3 人被抽中
  ↓
触发后生成世界待执行事件：
  {
    事件名: "张三寻找恩人",
    类型: "因果池触发",
    主角参与度: 0.7,  // ← 关键：因果池触发的事件也要标记参与度
    来源因果池: "主角救过的人",
    触发成员: ["张三"]
  }
  ↓
进入世界演变系统，按参与度 0.7 注入上下文：
  "附近动静：有人在打听你的下落"
```

**关键点**：
1. 因果池触发后生成的**仍然是世界事件**
2. 但这个世界事件的 `主角参与度` 较高（0.6-0.8）
3. 区分了"远处客观事件"（参与度 0.1-0.2）和"主角因果事件"（参与度 0.6-0.8）

### 3.4 实施建议

**短期**（叙事平静值系统实现时）：
- 只实现世界事件的"主角参与度"改造
- 暂不实现因果池机制

**中期**（世界事件改造后）：
- 实现因果池机制
- 因果池触发后生成带 `主角参与度` 字段的世界事件

**长期**（稳定后）：
- 考虑更复杂的因果池策略（如"关系强度随时间增强"）

## 五、风险评估

### 高风险项

1. **AI 服从率问题**
   - 即使格式说明强调，AI 仍可能漏打 `<情节事件>` 标签
   - 缓解：关键词软检测 + 监控漏标率

2. **参与度计算复杂性**
   - 如何自动判断一个事件的主角参与度？
   - 当前方案：由世界演变生成时手动标记 + 根据主角位置动态调整
   - 风险：可能需要 AI 辅助判断

3. **因果池触发概率调参**
   - `基础触发概率` 和 `时间衰减系数` 需要大量测试
   - 建议：先用保守值（基础 0.15，衰减 0.05），后续根据测试调整

### 中风险项

4. **快照序列化膨胀**
   - 因果池成员列表可能很长（100+ 人）
   - 缓解：定期清理低关系强度成员

5. **世界演变性能**
   - 每回合检测所有因果池 + 所有到期事件
   - 缓解：设置冷却回合，避免高频检测

### 低风险项

6. **旧存档兼容性**
   - 已有兜底逻辑，旧存档不会崩溃
   - 只是丢失叙事平静值数据和参与度数据

## 六、总结

### 6.1 修改计划 ver 6 评估结论

**设计逻辑：✅ 基本合理**
- 参数调整、架构边界、多标签策略、阈值文本设计均符合元叙事节奏系统的定位
- 需要补充 9 个实现细节，其中 3 个为 P0 阻塞项

### 6.2 世界事件改造结论

**必要性：✅ 高度必要**

**核心理由**：
1. 避免"世界围着主角转"（主角上山砍柴都被拉进主线）
2. 区分"远处客观事件"（因果很深）和"主角因果事件"（因果较浅）
3. 让世界演变系统能够真正独立运行

**改造方案**：
- 追加 `主角参与度` 字段（0.0~1.0）
- 按参与度分级注入（远处传闻/附近动静/当前事件）
- 约束 AI：参与度 < 0.3 的事件只能通过间接方式获知

### 6.3 因果池机制定位

**定位：独立系统，与世界事件协调**
- 因果池处理"批量个人因果链"（主角救了 20 人）
- 触发后生成带 `主角参与度` 的世界事件
- 不是世界事件的替代品，而是补充

**实施优先级**：
- 短期：只做世界事件改造（P1）
- 中期：实现因果池机制（P3）

### 6.4 最终建议

**分三步走**：
1. **阶段 0**（P0）：修复叙事平静值系统本身的实现细节
2. **阶段 1**（P1）：改造世界事件系统，追加主角参与度
3. **阶段 2-3**（P2-P3）：联动叙事平静值 + 实现因果池（可选）

**关键认识**：
- 叙事平静值系统是"呼吸提示器"，不能根治"世界围着主角转"
- 世界事件改造是根本解决方案，必须实施
- 因果池是锦上添花，解决批量因果管理问题

## 七、根因深挖：世界围着主角转的根源分析（2026-06-30）

> 本节整合了两轮分析：第一轮（7.1-7.5）定位六个叠加根因，第二轮（7.6-7.11）做完整链路排查与改造方案核对。详细改造方案已写入 `修改计划.md` 第六章。

### 7.1 链路排查结论（第一轮）

从**世界事件生成 → 世界事件注入主剧情 → 主剧情 AI 消费事件**的完整链路追踪，定位到六个叠加根因：

#### 根因 1：数据层缺距离标记（最核心）

`systemPromptBuilder.ts` 的 `构建世界状态文本()` 把 `待执行事件 / 进行中事件 / 已结算事件` 全部平铺注入主剧情。每个事件有 `关联地点[]`，但**没有任何字段告诉 AI：这个事件离主角多远？主角该直接遇到，还是只该听说？**

`关联地点[]` 只是字符串数组，LLM 需自己拿它和 `环境.大地点 > 中地点 > 小地点 > 具体地点` 做空间比对——LLM 极不擅长这种隐式空间推理，默认倾向是"既然注入了，就该和主角有关"。

#### 根因 2：主剧情 COT Step5 用"触发"措辞

`prompts/core/cot.ts` Step5 原文：
> 确认本回合是否存在适合**触发**的当前事件、群像镜头、镜头外余波或远端动静。

"触发"有"让它发生"的语义倾向。且没给"该上台 vs 只做背景"的判断标准。AI 自然解读成"该让它在主角面前发生了"。

#### 根因 3：防全知协议的反向激励（悖论）

`prompts/writing/antiOmniscient.ts`：锁定玩家视角，不得描写玩家不在场场景。

这意味着 AI 不能写"城东灭门案现场"（主角不在城东）。它可以写成"听说城东灭门了"——但写传闻链需要更多创造性努力（谁说的？在哪听到？怎么传过来？）。

LLM 的路径选择倾向是**阻力最小路径**：与其构思传闻传播链，不如让灭门案凶手"恰好路过"主角所在的西山——直接写成场景，不需要额外构思信息传递渠道。

**防全知协议本意是防止全知视角，但意外地产生了"把事件拉到主角身边"的副作用。**

#### 根因 4：世界演变补位规则偏向主角

`prompts/runtime/worldEvolution.ts` 与 `prompts/stats/world.ts`：补位优先"既有旧线 > 已成立人物下一段动作"。

而"既有旧线"和"已成立人物"几乎都是主角相关的——整个故事都围绕主角展开。当世界演变 AI 需补满 5 条进行中事件时，自然从主角身边的事开始补。**补位规则本身就把世界拉向主角。**

#### 根因 5：规划分析等权呈现全部事件

`prompts/runtime/planningAnalysis.ts`：规划分析 AI 拿到完整 `世界.待执行事件` 和 `世界.进行中事件`，所有事件等权呈现。当它决定"补承接"时，自然选和当前剧情最相关的——也就是主角相关的。没有任何指令告诉它"参与度 < 0.3 的事件不要拉进当前章任务"。

#### 根因 6：LLM 固有主角中心化倾向

LLM 通性，被以下设计放大：
- 整个系统提示词以主角为框架（`Step0: 先审计主角当前基底`）
- 角色状态、社交状态、任务约定全是主角视角
- `storyStyles/index.ts` 的"去魅主"只针对 NPC 对主角的态度，不涉及事件空间分布

### 7.2 深度排查：worldEvolutionCot.ts 的结构性问题（第二轮）

进一步排查 `worldEvolutionCot.ts` 发现三个更深层的问题：

#### 7.2.1 "当前"时间锚定

`worldEvolutionCot.ts` 中大量使用"当前"这一时间锚点。统计：164 行中约 18 行包含"当前"变体，12 行包含"本回合"。

**但经核对，多数"当前"指"当前世界状态"而非"主角当前状态"**（如"当前世界维护使用的是新结构"）。部分"主角"是禁止性规则（"主角不得进入活跃 NPC 列表"）。因此"49.4% 主角中心化"的统计存在偏差，不应作为大规模改写 COT 时间锚定语言的依据。

**真正需要修正的**是 Step0 的前台事实用途——见下一项。

#### 7.2.2 前台事实直接注入世界演变 COT

`worldEvolutionCot.ts` Step0 证据清单中包含"本回合前台已发生事实"。这与"背景演变不是台前跟拍"的设计声明存在张力。

**但完全删除前台事实会破坏世界状态一致性**：
- 主角杀了某 NPC → 世界演变需更新活跃 NPC 列表 → 需要前台事实
- 主角解决了某事件 → 世界演变需迁移事件状态 → 需要前台事实

**正确方案**（已写入修改计划 6.3.7）：保留前台事实作为承接依据，但追加"承接 vs 推导"区分：
- 合法用途：承接已发生后果（更新列表、迁移状态、刷新关系）
- 禁止用途：以主角行动为起点推导新事件方向

#### 7.2.3 负面表述的逆向效应

`worldEvolutionCot.ts` Step1 原文：
> 若当前信息仍停留在台前场景跟拍...就先排除出本链路

"先提到再否定"的表述让 AI 优先激活了应该避免的概念。**已采纳修正**（修改计划 6.3.7）：改为正向表述"只处理已转化为后台可持续状态的信息"。

### 7.3 核心洞察

**"世界围着主角转"不是世界演变 AI 的错，而是主剧情 AI 在消费世界事件时，没有足够的约束阻止它走阻力最小路径。**

世界演变生成的远处事件本身是正确的（城东灭门案客观发生），但到了主剧情 AI 手里，因为：
- 没有距离标记 → 不知道该听说还是该遇见
- "触发"措辞 → 鼓励让它发生
- 防全知锁定 → 写现场比写传闻容易
- 没有强拉禁令 → "恰好路过"不被禁止

→ 远处事件被拉到主角面前。

**因此改造重心应在主剧情消费侧（COT + 防全知 + 注入分档），而非只在生成侧（世界演变）。**

### 7.4 改造优先级

- **P1a（1-2 天）**：数据字段 + 主剧情注入分档 + COT Step5 + 防全知 5.1 节 + format 5.5 节。这五项是"立刻见效"的核心，阻断主剧情 AI 的强拉路径。
- **P1b（1-2 天）**：世界演变补位规则 + Step0 承接vs推导 + Step1 正向表述 + 规划分析参与度过滤。让生成侧和规划侧也尊重距离。
- **P2（1-2 天）**：与叙事平静值联动（推进远处事件）。

P1a 可与叙事平静值系统（阶段 0）并行或紧接着实施，因为它不依赖平淡值系统。

### 7.5 风险与缓解

1. **传闻链写不好 → 信息密度下降**：强制远处事件走传闻链后，AI 可能写出干瘪的"听说某某事"。缓解：COT Step5 补充"传闻层也要写出传播渠道（谁说的、在哪听到、榜文贴在哪）"，让传闻有画面感。
2. **参与度计算偏差**：世界演变 AI 可能误判参与度。缓解：参与度是建议非锁；主剧情 AI 可基于正文事实调整；规划分析允许在正文坐实后回写。
3. **旧存档中性值 0.5**：旧事件全标 0.5 = 附近。缓解：可接受——0.5 比"无标记默认当场"已是大改善。
4. **过度约束 → 主角长期无事可做**：缓解：世界演变提示词明确"主角约定事件→0.9当场"；参与度由 AI 标记，AI 会根据因果理解当场事件。
5. **主角主动去远处调查 → 事件仍标远处 → 矛盾**：缓解：正文事实坐实后，世界演变更新参与度。

### 7.6 改造方案完整性验证

### 7.7 与 Claude 分析的差异说明

Claude 的分析方向正确，但以下三项做了修正（详见修改计划 6.8）：

1. **不删除 worldEvolutionCot.ts Step0 的前台事实**：完全删除会破坏世界状态一致性。改为追加"承接 vs 推导"区分。
2. **不大规模改写 COT 时间锚定语言**：多数"当前"指世界状态而非主角状态，统计存在偏差。只采纳负面表述改正向。
3. **不用程序化规则覆盖 AI 参与度标记**：程序化规则无法处理复杂间接因果。改为 AI 标记为主 + 程序化兜底为辅。

### 7.8 副作用评估

详见修改计划 6.7。核心结论：
- 两系统协同：平淡值提示"该有新事了"，距离化约束"新事不一定在主角面前"——这是期望行为。
- 传闻链信息密度下降是主要风险，通过 COT Step5 补充"传播渠道"缓解。
- 旧存档中性值 0.5 可接受，新事件从改造上线起显式设定。
- 过度约束风险通过"主角约定事件→0.9当场"和"正文坐实后回写参与度"缓解。
